# LPE 0.5.1

`0.5.1` establishes `0.5.1-sql` as the canonical database schema while the
Outlook MAPI over HTTP cached-mode validation campaign continues across Mail,
Calendar, and Contacts.

## Database and update policy

- New deployments require an empty PostgreSQL database initialized from
  `crates/lpe-storage/sql/schema.sql` by
  `installation/debian-trixie/init-schema.sh`.
- The canonical schema version is exactly `0.5.1-sql`.
- `update-lpe.sh` supports exactly one in-place release transition: the late
  canonical physical form of `0.5.0-sql-v1` to `0.5.1-sql`.
- A read-only preflight rejects older physical variants carrying the same
  0.5.0 label before the service is stopped or the database is mutated.
- For the supported source, the updater applies the reviewed Outlook cache
  fidelity update, validates the complete target shape, then writes the new
  label. The SQL updates are forward-only, transactional, idempotent, and
  version-bounded.
- The label-only transition requires the private target-validation marker set
  by `update-lpe.sh`; operators rerun the updater rather than invoking that SQL
  file directly.
- Every other source label, including releases before 0.5, is also rejected
  before the service is stopped or the database is mutated.
- `schema.sql` remains the complete canonical source for every new database;
  migration SQL does not become a second schema definition or parallel state.
- `LPE_RESET_SCHEMA=true` remains an explicit destructive action for a
  disposable or intentionally rebuilt deployment.

## Outlook validation

- New `0.5.1` installations enable `LPE_AUTOCONFIG_MAPI_ENABLED=true` for
  capable Outlook clients; legacy `EXCH`, `EXPR`, RPC over HTTP, SOAP Exchange
  Autodiscover, and EWS remain independently controlled.
- Final cached-mode acceptance uses a fresh `0.5.1-sql` database and a clean
  Outlook profile/OST, then verifies Mail, Calendar, Contacts, synchronization,
  notifications, and reopen after Outlook and server restarts.
- The supported in-place migration preserves canonical rows for installations
  that intentionally retain their current test data; it does not replace the
  fresh-database acceptance baseline.

The historical `0.5.0` release policy and milestones remain recorded in
`docs/releases/0.5.0.md`.
