# LPE 0.4.0

`0.4.0` starts from a fresh database schema and supports bounded, documented
SQL compatibility updates for initialized `0.4` deployments.

## Upgrade Policy

- New core `LPE` databases must be empty before applying the `0.4.0-sql-v2`
  schema with `installation/debian-trixie/init-schema.sh`.
- Existing initialized `0.4` databases can be advanced by
  `installation/debian-trixie/update-lpe.sh` when the repository includes a
  documented, idempotent SQL compatibility patch for the current schema version.
- `update-lpe.sh` checks the installed schema version before running any SQL
  compatibility patch, then rebuilds and redeploys code and web assets.
- Use `LPE_RESET_SCHEMA=true` with `init-schema.sh` only for an intentional
  destructive reset of a disposable or explicitly rebuilt deployment.

## Operator Notes

- After initialization, `check-lpe.sh` verifies that the installed database
  reports schema version `0.4.0-sql-v2` and includes any documented compatibility
  columns required by the current code.
- Current compatibility patch: `update-lpe.sh` makes
  `mapi_navigation_shortcuts.target_folder_id` nullable for Common Views group
  header rows and adds `group_header_id` plus `group_name` for Outlook
  `WunderBar` shortcut group/header persistence.
- The same patch adds the canonical recoverable-items store, account/mailbox
  retention and legal-hold columns, and change-log/tombstone constraints needed
  before MAPI dumpster ROP behavior is exposed.
- A missing or weak `LPE_INTEGRATION_SHARED_SECRET` degrades readiness and
  pauses the outbound bridge worker, but the core administration API still
  starts so operators can sign in and repair configuration.
