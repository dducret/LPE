# LPE 0.5.0

`0.5.0` establishes a new product and database baseline after the first
successful real Outlook MAPI over HTTP profile, cached-mode, receive, reply,
send, notification, and restart milestone.

## Database and update policy

- New deployments require an empty PostgreSQL database initialized from
  `crates/lpe-storage/sql/schema.sql` by
  `installation/debian-trixie/init-schema.sh`.
- The schema version is `0.5.0-sql-v1`.
- Databases from releases before 0.5.0 are not upgraded in place.
- `update-lpe.sh` accepts only the exact current schema version and performs no
  SQL compatibility mutation.
- The current physical baseline includes `mapi_special_folder_aliases`, an
  account-scoped, many-to-one mapping from bounded Outlook client
  FID/SourceKey aliases to canonical special-folder FIDs, plus the distinct
  server CN required by hierarchy-upload state. It stores protocol identity
  only; Calendar, Contacts, mailbox content, FAI, and rights remain in
  their canonical tables.
- A database created from an earlier 0.5.0 checkout without that table is
  incomplete even when labeled `0.5.0-sql-v1`. It must be recreated from an
  empty database or intentionally reset; there is no in-place compatibility
  update.
- `LPE_RESET_SCHEMA=true` remains an explicit destructive action for a
  disposable or intentionally rebuilt deployment.

## Outlook

- New installations enable `LPE_AUTOCONFIG_MAPI_ENABLED=true`.
- A capable Outlook client receives authenticated `mapiHttp` EMSMDB and NSPI
  endpoints without the former second MAPI publication flag.
- `LPE_AUTOCONFIG_OUTLOOK_INTEROP_GATE_PASSED` remains a legacy `EXPR` / RPC
  over HTTP gate and does not control MAPI over HTTP.
- Legacy `EXCH`, `EXPR`, RPC over HTTP, SOAP Exchange Autodiscover, and EWS
  remain independently configurable.
- Mailbox startup, cached-mode reopen, Inbox display, send/reply, canonical
  `Sent`, read state, notifications, and accented MIME body ingestion have
  passed the current real-client investigation.
- Outlook hierarchy upload now preserves validated client-reserved special-folder
  FIDs as durable account-scoped aliases across EMSMDB sessions and server
  restarts. The mapping is many-to-one so separate profiles/OST replicas can
  map their own FIDs to the same canonical Calendar, Contacts, Deleted Items,
  or other special folder without creating a shadow folder.
- A hierarchy import allocates a server CN distinct from the imported FID.
  Upload collector transfer state merges the applicable initial and newly
  assigned CN sets, ignores `MetaTagIdsetGiven`, and does not return it.
- Calendar and Contacts remain implemented protocol surfaces. Their final
  cached-mode acceptance must use the newly initialized database and a new
  Outlook profile/OST, then verify reopen after an Outlook and server restart;
  evidence from a profile built against the incomplete schema is not valid.

## Installation

- `update-lpe.sh` does not carry compatibility SQL or mutate the schema.
- PostgreSQL provisioning and canonical schema initialization are reached only
  through the installation orchestrator.
- `install-lpe.sh` remains the single installation orchestrator. It invokes
  the focused local PostgreSQL provisioner and the canonical schema initializer
  when appropriate.
- `init-schema.sh`, `check-lpe.sh`, and `update-lpe.sh` validate the special-folder
  alias columns, bounded FID and 22-byte SourceKey checks, account foreign key,
  alias/SourceKey uniqueness, and the required absence of canonical-FID
  uniqueness. They diagnose a damaged current database but do not direct
  operators to an unsupported compatibility update.

## Protocol references

MAPI over HTTP publication continues to use:

- `[MS-OXCMAPIHTTP]` sections 2.2.3 and 2.2.4 for MAPI/HTTP request and response
  transport behavior.
- `[MS-OXDSCLI]` sections 2.2.2.1, 2.2.4.1.1.2.4.30, and 4.4 for
  `X-MapiHttpCapability` negotiation and the MapiHttp Autodiscover response.
- `[MS-OXDISCO]` sections 2.2.3 and 3.2.3.5 for HTTP publication and discovery
  behavior.
- [[MS-OXCFXICS] section 2.2.3.2.4.1.1](https://learn.microsoft.com/en-us/openspecs/exchange_server_protocols/ms-oxcfxics/7fa21d88-71ba-4cfe-8af5-4d7902489e5f),
  [[MS-OXCROPS] section 2.2.13.7.1](https://learn.microsoft.com/en-us/openspecs/exchange_server_protocols/ms-oxcrops/8588e7dd-4732-4d3e-bf28-82bbdf0b4d04),
  and
  [[MS-OXCDATA] section 2.11.1](https://learn.microsoft.com/en-us/openspecs/exchange_server_protocols/ms-oxcdata/0c77892e-288e-435a-9c49-be1c20c7afdb)
  for the one-byte, 8-bit
  `RopSynchronizationOpenCollector.IsContentsCollector` Boolean: `0` is a
  hierarchy upload collector and `1` is a contents upload collector.
- [[MS-OXCFXICS] section 3.2.5.2.1](https://learn.microsoft.com/en-us/openspecs/exchange_server_protocols/ms-oxcfxics/696ec085-cfb4-451c-a576-7cdc81b1550a)
  and sections 2.2.3.2.3.1, 2.2.4.4, and 3.2.5.9.3.1 plus
  `[MS-OXCROPS]` sections 2.2.13.8.1 and 2.2.13.8.2 for upload checkpoint state
  and omission of `MetaTagIdsetGiven`.
- [[MS-OXCFXICS] section 3.1.5.3](https://learn.microsoft.com/en-us/openspecs/exchange_server_protocols/ms-oxcfxics/b03168dc-6ad4-4747-b1b5-c95fd436dfce)
  for a server CN distinct from imported object identity, and
  [[MS-OXCFXICS] section 2.2.3.2.4.3.1](https://learn.microsoft.com/en-us/openspecs/exchange_server_protocols/ms-oxcfxics/9d5d9d68-775d-4ede-a14c-119bc54a6327),
  sections 2.2.3.2.4.7.2, 3.2.5.9.4.3, 3.3.5.2.1, and 3.3.5.8.12, and
  [[MS-OXCFXICS] section 3.3.5.8.8](https://learn.microsoft.com/en-us/openspecs/exchange_server_protocols/ms-oxcfxics/f3fab904-bb7d-4cf3-bbd6-65d4a34b67d2)
  for hierarchy import, SourceKey/FID handling, and client-reserved identifiers.

Exact source versions, official Microsoft URLs, cache metadata, hashes, and
sections remain recorded in `docs/microsoft/protocol-sources.toml`.
