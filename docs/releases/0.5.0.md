# LPE 0.5.0

`0.5.0` establishes a new product and database baseline after the first
successful real Outlook MAPI over HTTP profile, cached-mode, receive, reply,
send, notification, and restart milestone.

## Database and update policy

- New deployments require an empty PostgreSQL database initialized from
  `crates/lpe-storage/sql/schema.sql` by
  `installation/debian-trixie/init-schema.sh`.
- The schema version is `0.5.0-sql-v1`.
- Databases from releases before 0.5.0 are not upgraded in place.
- `update-lpe.sh` accepts only the exact current schema version and performs no
  SQL compatibility mutation.
- `LPE_RESET_SCHEMA=true` remains an explicit destructive action for a
  disposable or intentionally rebuilt deployment.

## Outlook

- New installations enable `LPE_AUTOCONFIG_MAPI_ENABLED=true`.
- A capable Outlook client receives authenticated `mapiHttp` EMSMDB and NSPI
  endpoints without the former second MAPI publication flag.
- `LPE_AUTOCONFIG_OUTLOOK_INTEROP_GATE_PASSED` remains a legacy `EXPR` / RPC
  over HTTP gate and does not control MAPI over HTTP.
- Legacy `EXCH`, `EXPR`, RPC over HTTP, SOAP Exchange Autodiscover, and EWS
  remain independently configurable.
- Mailbox startup, cached-mode reopen, Inbox display, send/reply, canonical
  `Sent`, read state, notifications, and accented MIME body ingestion have
  passed the current real-client investigation.
- Calendar and Contacts remain implemented protocol surfaces, but should be
  included in the next manual 0.5.0 acceptance pass because they were not
  re-exercised in the final mail-focused run.

## Installation

- `update-lpe.sh` does not carry compatibility SQL or mutate the schema.
- PostgreSQL provisioning and canonical schema initialization are reached only
  through the installation orchestrator.
- `install-lpe.sh` remains the single installation orchestrator. It invokes
  the focused local PostgreSQL provisioner and the canonical schema initializer
  when appropriate.
- Schema shape checks remain in `check-lpe.sh`; they diagnose a damaged current
  database but no longer direct operators to an unsupported compatibility
  update.

## Protocol references

MAPI over HTTP publication continues to use:

- `[MS-OXCMAPIHTTP]` sections 2.2.3 and 2.2.4 for MAPI/HTTP request and response
  transport behavior.
- `[MS-OXDSCLI]` sections 2.2.2.1, 2.2.4.1.1.2.4.30, and 4.4 for
  `X-MapiHttpCapability` negotiation and the MapiHttp Autodiscover response.
- `[MS-OXDISCO]` sections 2.2.3 and 3.2.3.5 for HTTP publication and discovery
  behavior.

Exact source versions, official Microsoft URLs, cache metadata, hashes, and
sections remain recorded in `docs/microsoft/protocol-sources.toml`.
