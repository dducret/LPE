# EWS Audit - 2026-06-02

## Scope

This audit compares the current `LPE` working tree against Microsoft's EWS operation catalog:

- <https://learn.microsoft.com/en-us/exchange/client-developer/web-service-reference/ews-operations-in-exchange>
- <https://learn.microsoft.com/en-us/exchange/client-developer/web-service-reference/ews-xml-elements-in-exchange>

It is a static repository audit plus local test verification. It is not a live Outlook, Exchange, Microsoft Remote Connectivity Analyzer, or Exchange Online compatibility run.

## Assumptions and Success Criteria

Assumptions:

- "EWS compatible" means compatibility with Microsoft's documented SOAP EWS operation surface, not only the subset needed by current Outlook profile creation.
- `LPE` should keep EWS bounded to canonical mailbox, contacts, calendar, tasks, submission, public-folder, rights, and synchronization state. EWS must not introduce parallel Outbox, Sent, mailbox, reminder, delegate, rule, or user-configuration state.
- SQL sufficiency means PostgreSQL has canonical durable state for the behavior, not that every Microsoft operation is already exposed through EWS.

Success criteria for this audit:

1. Recheck every EWS dispatcher operation currently visible in `crates/lpe-exchange/src/service.rs`.
2. Compare the visible operation surface with Microsoft's operation catalog.
3. Identify visible differences and implemented workarounds.
4. Verify whether implemented EWS operations route to canonical APIs/storage or stateless compatibility responses.
5. Verify whether SQL contains the data needed for the implemented bounded EWS surface and for obvious remaining high-value parity gaps.

## Summary

`LPE` still implements bounded EWS compatibility rather than full Microsoft Exchange Web Services compatibility, but the implemented surface has grown substantially since the previous audit snapshot.

Newly implemented dispatcher paths since the last audit include:

- Folder operations: `CreateFolderPath`, `CopyFolder`, `EmptyFolder`, `MoveFolder`, `UpdateFolder`.
- Mailbox item operation: `MarkAllItemsAsRead`.
- Conversation operations: `FindConversation`, `GetConversationItems`, `ApplyConversationAction`.
- Utility operations: `ConvertId`, `ExpandDL`, `GetMailTips`, `GetServiceConfiguration`, `GetUserPhoto`, `GetPasswordExpirationDate`, `MarkAsJunk`.
- Retention operation: `GetUserRetentionPolicyTags`.
- Sharing operations: `GetSharingMetadata`, `GetSharingFolder`, `RefreshSharingFolder`, and `CreateItem` with `AcceptSharingInvitation`.
- eDiscovery and transfer operations: `GetDiscoverySearchConfiguration`, `GetSearchableMailboxes`, `SearchMailboxes`, `GetHoldOnMailboxes`, `SetHoldOnMailboxes`, `GetNonIndexableItemDetails`, `GetNonIndexableItemStatistics`, `UploadItems`, and `ExportItems`.
- Delegate operations: `AddDelegate`, `GetDelegate`, `UpdateDelegate`, `RemoveDelegate`.
- User-configuration operations: `CreateUserConfiguration`, `GetUserConfiguration`, `UpdateUserConfiguration`, `DeleteUserConfiguration`.

Explicit unsupported dispatcher paths are now limited to `FindPeople`.

Estimated current coverage:

- Microsoft EWS operation-name coverage: about 73%, or 69 implemented operation names out of roughly 94 unique Microsoft-documented EWS operation names.
- Full behavioral/schema compatibility: approximately 45-50%. The implemented operations are canonical and increasingly broad, but many remain deliberately partial compared with Exchange semantics.
- LPE's documented bounded EWS surface: implemented for the audited dispatcher paths.

## Visible Differences and Workarounds

Current EWS compatibility behavior differs visibly from Exchange in these ways:

- The adapter has two case-compatible EWS routes: `/EWS/Exchange.asmx` and `/ews/exchange.asmx`.
- EWS authentication uses the shared mailbox account authentication path and returns an EWS Basic challenge when needed.
- Operation detection is based on the first SOAP body operation element name; it is not full WSDL/XSD validation.
- Unknown or out-of-scope operation names return parseable EWS error payloads instead of raw transport failures.
- EWS ids are LPE canonical prefixed identifiers. `ConvertId` adds deterministic stateless opaque alternate ids instead of Exchange-generated identity rows.
- `FindFolder`, `GetFolder`, `SyncFolderHierarchy`, `FindItem`, `GetItem`, `SyncFolderItems`, `CreateItem`, `UpdateItem`, `DeleteItem`, `MoveItem`, `CopyItem`, `CreateFolder`, `CreateFolderPath`, `CopyFolder`, `EmptyFolder`, `UpdateFolder`, and `DeleteFolder` include bounded public-folder behavior over canonical public-folder tables.
- Public-folder item move is implemented as canonical clone into target folder plus delete of the original item.
- Public-folder folder move is explicitly rejected because canonical public-folder reparenting is not yet implemented.
- Public-folder item support is bounded to post-like items; arbitrary Exchange item classes and recipient-bearing public-folder conversions remain out of scope.
- `SyncFolderItems` uses synthetic sync-state strings over current visible state rather than durable Exchange sync cursors.
- Pull and bounded streaming notifications replay canonical `mail_change_log` state through shared notification polling; they are not held in EWS-local memory or SQL subscription rows.
- `GetEvents` returns status when no canonical change exists and parseable `ErrorInvalidWatermark` when the requested cursor has expired from retention.
- `GetStreamingEvents` is an immediate bounded response rather than a full long-held Exchange streaming subscription session.
- `GetServerTimeZones` is a small static compatibility response.
- OOF maps to canonical Sieve vacation scripts and LPE metadata comments, not an Exchange OOF table.
- `CreateAttachment` validates bytes with Magika before storage and accepts only `FileAttachment`.
- `CreateItem` send dispositions and `SendItem` use canonical LPE submission and authoritative Sent visibility; there is no EWS/client SMTP path.
- `SendItem` submits existing canonical draft ids, preserves protected `Bcc` in canonical submission/Sent state, and does not create EWS-local Outbox or Sent state.
- `GetInboxRules` projects canonical Sieve-backed mailbox rules only.
- `UpdateInboxRules` creates, updates, and deletes only bounded rules that map safely to canonical Sieve behavior. Exchange rule blobs, client-only rules, and unsupported deferred-action data return parseable EWS errors without Sieve side effects.
- `GetReminders` returns computed canonical calendar/task/message reminder views.
- `PerformReminderAction` dismisses canonical calendar/task reminders through `reminder_occurrence_dismissals` and snoozes them by updating canonical reminder time/reset state; it does not create EWS reminder rows.
- `GetRooms` projects tenant-visible canonical room and equipment directory accounts only; hidden accounts, person accounts, and other tenants are excluded.
- `GetRoomLists` returns LPE's computed tenant room/resource list when visible room or equipment accounts exist. Arbitrary Exchange room-list membership is not implemented.
- Delegate operations mutate canonical grants, sender rights, and delegate preferences. They do not create Exchange-only delegate folder state.
- User-configuration operations persist bounded payloads in `account_client_configurations`; there is no Exchange roaming-config side table.
- `GetMailTips` uses canonical directory visibility and Sieve vacation projection; it is not a full Exchange policy-tip/mailbox-quota/mail-flow tip engine.
- `GetServiceConfiguration` exposes only the existing bounded MailTips capability. Unified Messaging, Protection Rules, Policy Tips, and unknown service-configuration probes return per-configuration EWS `ErrorInvalidOperation` response messages because LPE does not currently have canonical engines for those Exchange service settings.
- `GetUserRetentionPolicyTags` returns only active same-tenant tags that are visible or assigned as the authenticated account's default tag. `OptedInto` is derived from `account_retention_policy_assignments.default_tag_id`; full Exchange retention-policy membership and managed-folder behavior remain out of scope.
- eDiscovery operations route through canonical compliance/search tables. Search results come from `mail_search_documents`, which excludes protected `Bcc`; hold operations mutate canonical compliance hold rows and account litigation-hold flags; non-indexable responses project `non_indexable_item_reports`.
- `UploadItems` and `ExportItems` record canonical transfer jobs and entries for EWS import/export requests. They do not implement full Exchange item byte-stream conversion or protocol-local transfer storage.
- Sharing operations are bounded to canonical same-tenant contact/calendar grants. `AcceptSharingInvitation` creates or updates a canonical grant for the authenticated account, `GetSharingFolder` and `RefreshSharingFolder` only expose already accessible shared contact/calendar collections, and `GetSharingMetadata` emits shareable owned collection metadata without Exchange tokens or federation state.
- Conversation operations use the existing lightweight `mailbox_messages.thread_id` value as bounded conversation identity. This is sufficient for same-account message grouping, item fetch, and one-shot read/move/delete actions over current canonical messages, but it is not full Exchange conversation identity: there is no first-class `threads` table, conversation lifecycle, recursive conversation index, historical deleted-item membership, or durable "always apply to future messages" rule created by EWS.
- `DeleteItem` maps Exchange delete types onto canonical hard delete or Trash move behavior.

The schema is adequate for the currently implemented bounded EWS surface. It also contains canonical data models for several high-value gaps that are still not EWS-wired, including mail apps and unified messaging. Those are storage-ready but not SOAP-compatible until dispatcher handlers, API methods, response shapes, and client-visible tests exist.

Remaining full-parity limits are primarily behavior and protocol semantics, not raw SQL table absence:

- Exchange managed/archive/search folder behavior is not fully modeled through EWS.
- Push/streaming subscription affinity is bounded to canonical replay, not Exchange long-lived subscription semantics.
- Room lists are computed from tenant room/resource accounts, not stored as arbitrary Exchange distribution-room-list objects.
- Mail app, UM, UCS, message-tracking, and policy-tip operation handlers are not wired even where canonical schema exists.
- `ConvertId` is stateless and deterministic for supported LPE ids; it is not full Exchange id compatibility with every historical id format.

## Conclusions

1. The EWS audit result has changed materially: top-level operation-name coverage is now about 73%, up from the previous 32-operation snapshot.
2. The newly wired operations route through canonical LPE APIs/storage or stateless compatibility codecs. I did not find EWS-local Sent, Outbox, reminder, delegate, rule, user-configuration, or subscription state competing with canonical state.
3. SQL is sufficient for the bounded implemented EWS behavior and includes canonical state for several remaining parity gaps.
4. `LPE` is still not 100% Microsoft EWS compatible because many remaining operations are absent, and several implemented operations still have bounded Exchange semantics.

## Highest-Value Remaining EWS Gaps

Priority for practical Outlook/native-client compatibility:

1. Implement mail app operations only if Outlook add-ins are in scope for the release.
2. Add message tracking, UM, and UCS operations only when their canonical APIs are complete enough to avoid trace-specific SOAP shims.
3. Promote conversation behavior from lightweight `thread_id` grouping to a first-class thread table only when JMAP `Thread/changes`, MAPI conversation IDs, or thread lifecycle semantics require it.
