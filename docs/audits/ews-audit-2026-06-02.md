# EWS Audit - 2026-06-02

## Scope

This audit compares the current `LPE` working tree against Microsoft's EWS operation catalog:

- <https://learn.microsoft.com/en-us/exchange/client-developer/web-service-reference/ews-operations-in-exchange>
- <https://learn.microsoft.com/en-us/exchange/client-developer/web-service-reference/ews-xml-elements-in-exchange>

It is a static repository audit plus local test verification. It is not a live Outlook, Exchange, Microsoft Remote Connectivity Analyzer, or Exchange Online compatibility run.

## Summary

`LPE` still implements bounded EWS compatibility rather than full Microsoft Exchange Web Services compatibility.

The important change since the 2026-05-30 audit is public-folder depth. The dispatcher still exposes the same top-level supported EWS operation names, but the existing folder and item operations now also project and mutate canonical public-folder folders and post-like public-folder items through `public-folder:{uuid}` and `public-folder-item:{uuid}` identifiers.

Estimated coverage:

- Microsoft EWS operation-name coverage: about 26%, or 24 implemented operation names out of roughly 94 unique Microsoft-documented EWS operation names.
- Full behavioral/schema compatibility: still lower, approximately 18-22%, because several implemented operations remain bounded and many Exchange behavior areas are absent.
- LPE's documented bounded EWS surface: implemented for the audited dispatcher paths.

Verification run:

- `cargo test -p lpe-exchange`: passed, 857 tests.
- `cargo test -p lpe-storage --test runtime_schema_drift`: passed, 1 test.

## Visible Differences and Workarounds

Current EWS compatibility behavior differs visibly from Exchange in these ways:

- The adapter has two case-compatible EWS routes: `/EWS/Exchange.asmx` and `/ews/exchange.asmx`.
- EWS authentication uses the shared mailbox account authentication path and returns an EWS Basic challenge when needed.
- Operation detection is based on the first SOAP body operation element name; it is not full WSDL/XSD validation.
- Unknown or out-of-scope operation names return parseable EWS error payloads instead of raw transport failures.
- EWS ids are LPE canonical prefixed identifiers, not Exchange opaque ids.
- `FindFolder`, `GetFolder`, `SyncFolderHierarchy`, `FindItem`, `GetItem`, `SyncFolderItems`, `CreateItem`, `UpdateItem`, `DeleteItem`, `MoveItem`, `CopyItem`, `CreateFolder`, and `DeleteFolder` now include bounded public-folder behavior over canonical public-folder tables.
- Public-folder item move is implemented as canonical clone into target folder plus delete of the original item.
- Public-folder item support is bounded to post-like items; arbitrary Exchange item classes and recipient-bearing public-folder conversions remain out of scope.
- `SyncFolderItems` uses synthetic sync-state strings over current visible state rather than durable Exchange sync cursors.
- Pull and bounded streaming notifications replay canonical `mail_change_log` state through the shared notification poll API; they are not held in EWS-local memory or SQL tables.
- `GetEvents` returns status when no canonical change exists and parseable `ErrorInvalidWatermark` when the requested cursor has expired from retention.
- `GetServerTimeZones` is a small static compatibility response.
- OOF maps to canonical Sieve vacation scripts and LPE metadata comments, not an Exchange OOF table.
- `CreateAttachment` validates bytes with Magika before storage and accepts only `FileAttachment`.
- `CreateItem` send dispositions use canonical LPE submission and authoritative Sent visibility; there is no EWS/client SMTP path.
- `SendItem` submits existing canonical draft ids through LPE submission, preserves authoritative canonical Sent visibility, keeps protected `Bcc` in canonical submission/Sent state, and does not create EWS-local Outbox or Sent state.
- `GetInboxRules` projects canonical Sieve-backed mailbox rules only.
- `UpdateInboxRules` creates, updates, and deletes only bounded rules that map safely to canonical Sieve behavior; Exchange rule blobs, client-only rules, and deferred-action data return parseable EWS errors without Sieve side effects.
- `GetReminders` returns computed canonical calendar/task reminder views.
- `PerformReminderAction` dismisses canonical calendar/task reminders through `reminder_occurrence_dismissals` and snoozes them by updating canonical reminder time/reset state; it does not create EWS reminder rows.
- `GetRooms` projects tenant-visible canonical room and equipment directory accounts only; hidden accounts, person accounts, and other tenants are excluded.
- `GetRoomLists` returns LPE's computed tenant room/resource list when visible room or equipment accounts exist. Arbitrary Exchange room-list membership is not implemented.
- `DeleteItem` maps Exchange delete types onto canonical hard delete or Trash move behavior.

## SQL Data Sufficiency

The current SQL schema contains the durable state needed for the implemented bounded EWS behavior:

| EWS capability | SQL support |
| --- | --- |
| Auth, tenant/domain/account identity | `tenants`, `domains`, `accounts`, `account_email_addresses`, `account_credentials`, `account_sessions`, app-password/OIDC tables |
| Mail folders/messages | `mailboxes`, `mailbox_messages`, `messages`, `message_headers`, `message_recipients`, `protected_bcc_recipients`, `message_bodies`, `mail_search_documents` |
| Attachments and MIME/blob metadata | `blobs`, `blob_placements`, `mime_parts`, `attachments`, `attachment_extraction_jobs`, `attachment_texts`, `calendar_event_attachments` |
| Contacts | `contact_books`, `contacts`, `contact_book_grants` |
| Calendar/free-busy | `calendars`, `calendar_events`, `calendar_grants` |
| Tasks/reminders | `task_lists`, `tasks`, `reminder_occurrence_dismissals` |
| OOF/vacation | `sieve_scripts`, `sieve_vacation_responses` |
| Submission/Sent | `account_identities`, `sender_rights`, `submission_queue`, `submission_recipients`, `submission_events`, canonical mail tables |
| Public folders | `public_folder_trees`, `public_folders`, `public_folder_items`, `public_folder_permissions`, `public_folder_replicas`, `public_folder_per_user_state` |
| Cross-protocol changes/deletes | `account_sync_state`, `mail_change_log`, `tombstones`, `canonical_change_journal`, `recoverable_items` |
| Audit | `audit_events` |

The schema is not sufficient for full Microsoft EWS parity. Missing or insufficient durable models include eDiscovery/hold reports, Exchange user-configuration blobs, mail apps, Unified Messaging, Unified Contact Store/IM groups, full room-list objects, message tracking reports, Exchange push/streaming affinity subscriptions, full EWS delegate objects, retention policy tags, full managed/archive folder semantics, full EWS bulk import/export stream state, and full Exchange opaque id conversion state for every EWS object.

## Conclusions

1. The EWS audit result changed in depth, not only in top-level operation count: public-folder support is now wired through existing EWS folder/item operations, and `SendItem`, `GetInboxRules`, `UpdateInboxRules`, `GetReminders`, `PerformReminderAction`, `GetRooms`, and `GetRoomLists` are now bounded implemented dispatcher paths.
2. Broader operation-count updates should be taken from the current interoperability matrix rather than this historical audit snapshot.
3. Several implemented operation names remain partial compared with Exchange semantics.
4. All currently dispatched EWS operations route to canonical storage APIs or stateless compatibility responses.
5. The SQL database is adequate for the bounded EWS surface, including the new public-folder backing tables.
6. The SQL database and EWS dispatcher are still not close to 100% Microsoft EWS parity.

## Highest-Value Remaining EWS Gaps

Priority for practical Outlook/native-client compatibility:

1. Widen `SendItem` only if real clients require additional Exchange disposition options, while preserving canonical submission.
2. Widen `UpdateInboxRules` only for additional shapes that map safely to canonical Sieve behavior.
3. Widen reminder actions only when they map to canonical reminder fields.
4. Add explicit canonical room-list membership only if clients require more than LPE's computed tenant room/resource list.
5. Harden notification long-poll/streaming affinity only if Outlook/EWS clients require more than bounded canonical replay.
6. `CreateUserConfiguration`, `UpdateUserConfiguration`, and `DeleteUserConfiguration` if Outlook/EWS clients require profile-side blobs.
