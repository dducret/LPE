# EWS Audit - 2026-06-02

## Scope

This audit compares the current `LPE` working tree against Microsoft's EWS operation catalog:

- <https://learn.microsoft.com/en-us/exchange/client-developer/web-service-reference/ews-operations-in-exchange>
- <https://learn.microsoft.com/en-us/exchange/client-developer/web-service-reference/ews-xml-elements-in-exchange>

It is a static repository audit plus local test verification. It is not a live Outlook, Exchange, Microsoft Remote Connectivity Analyzer, or Exchange Online compatibility run.

## Assumptions and Success Criteria

Assumptions:

- "EWS compatible" means compatibility with Microsoft's documented SOAP EWS operation surface, not only the subset needed by current Outlook profile creation.
- `LPE` should keep EWS bounded to canonical mailbox, contacts, calendar, tasks, submission, public-folder, rights, and synchronization state. EWS must not introduce parallel Outbox, Sent, mailbox, or reminder state.
- SQL sufficiency means PostgreSQL has canonical durable state for the behavior, not that every Microsoft operation is already exposed through EWS.

Success criteria for this audit:

1. Recheck every EWS dispatcher operation currently visible in `crates/lpe-exchange/src/service.rs`.
2. Compare the visible operation surface with Microsoft's operation catalog.
3. Identify visible differences and implemented workarounds.
4. Verify whether implemented EWS operations route to canonical APIs/storage or stateless compatibility responses.
5. Verify whether SQL contains the data needed for the implemented bounded EWS surface and for obvious remaining high-value parity gaps.

## Summary

`LPE` still implements bounded EWS compatibility rather than full Microsoft Exchange Web Services compatibility.

The important changes since the earlier 2026-06-02 audit snapshot are top-level EWS surface growth and more durable schema support:

- Newly implemented dispatcher paths: `SendItem`, `MarkAllItemsAsRead`, `GetInboxRules`, `UpdateInboxRules`, `GetReminders`, `PerformReminderAction`, `ConvertId`, `GetMailTips`, `GetRooms`, `GetRoomLists`, `GetStreamingEvents`, `CreateFolderPath`, `CopyFolder`, `EmptyFolder`, `MoveFolder`, `UpdateFolder`, `CreateUserConfiguration`, `GetUserConfiguration`, `UpdateUserConfiguration`, `DeleteUserConfiguration`, `AddDelegate`, `GetDelegate`, `UpdateDelegate`, and `RemoveDelegate`.
- Explicit unsupported dispatcher paths are now limited to `FindPeople`, `ExpandDL`, `GetSharingMetadata`, and `GetSharingFolder`.
- SQL now includes additional EWS/MAPI parity state for public folders, reminder occurrence dismissals, delegate preferences, retention policy tags, mail apps, and unified messaging call tracking.

Automated compatibility-gate coverage:

- Microsoft EWS operation-name coverage: 100%, or 96 documented operation names with either a behavior test or an explicit unsupported SOAP test.
- Implemented behavioral operation coverage: 50.0%, or 48 behavior-tested operation names out of 96 documented operation names tracked by the parity matrix.
- Explicit unsupported operation coverage: 50.0%, or 48 documented operation names with passing unsupported SOAP assertions.
- Full behavioral/schema compatibility: approximately 25-30%. The implemented operations are canonical and useful, but many are deliberately partial compared with Exchange semantics.
- LPE's documented bounded EWS surface: implemented for the audited dispatcher paths.

## Visible Differences and Workarounds

Current EWS compatibility behavior differs visibly from Exchange in these ways:

- The adapter has two case-compatible EWS routes: `/EWS/Exchange.asmx` and `/ews/exchange.asmx`.
- EWS authentication uses the shared mailbox account authentication path and returns an EWS Basic challenge when needed.
- Operation detection is based on the first SOAP body operation element name; it is not full WSDL/XSD validation.
- Unknown or out-of-scope operation names return parseable EWS error payloads instead of raw transport failures.
- EWS ids are LPE canonical prefixed identifiers, not Exchange opaque ids.
- `ConvertId` opaque alternate ids are deterministic encodings of LPE canonical ids: `LPEEWS1.` plus URL-safe base64 of `lpeews:v1:<canonical-id>`, with `HexEntryId` represented as uppercase hex of that same opaque token.
- `FindFolder`, `GetFolder`, `SyncFolderHierarchy`, `FindItem`, `GetItem`, `SyncFolderItems`, `CreateItem`, `UpdateItem`, `DeleteItem`, `MoveItem`, `CopyItem`, `CreateFolder`, `CreateFolderPath`, `CopyFolder`, `EmptyFolder`, `UpdateFolder`, and `DeleteFolder` include bounded public-folder behavior over canonical public-folder tables. Public-folder `MoveFolder` remains explicitly rejected until canonical public-folder reparenting exists.
- Public-folder item move is implemented as canonical clone into target folder plus delete of the original item.
- Public-folder item support is bounded to post-like items; arbitrary Exchange item classes and recipient-bearing public-folder conversions remain out of scope.
- `SyncFolderItems` uses synthetic sync-state strings over current visible state rather than durable Exchange sync cursors.
- Pull and bounded streaming notifications replay canonical `mail_change_log` state through shared notification polling; they are not held in EWS-local memory or SQL subscription rows.
- `GetEvents` returns status when no canonical change exists and parseable `ErrorInvalidWatermark` when the requested cursor has expired from retention.
- `GetStreamingEvents` is an immediate bounded response rather than a full long-held Exchange streaming subscription session.
- `GetServerTimeZones` is a small static compatibility response.
- OOF maps to canonical Sieve vacation scripts and LPE metadata comments, not an Exchange OOF table.
- `CreateAttachment` validates bytes with Magika before storage and accepts only `FileAttachment`.
- `CreateItem` send dispositions and `SendItem` use canonical LPE submission and authoritative Sent visibility; there is no EWS/client SMTP path.
- `SendItem` submits existing canonical draft ids, preserves protected `Bcc` in canonical submission/Sent state, and does not create EWS-local Outbox or Sent state.
- `GetInboxRules` projects canonical Sieve-backed mailbox rules only.
- `UpdateInboxRules` creates, updates, and deletes only bounded rules that map safely to canonical Sieve behavior. Exchange rule blobs, client-only rules, and unsupported deferred-action data return parseable EWS errors without Sieve side effects.
- `MarkAllItemsAsRead` bulk-updates canonical mailbox message read state through message flag updates; public-folder per-user read state is not bulk-mutated through this EWS operation yet.
- `GetMailTips` projects invalid-recipient and OOF tips from canonical address-book and Sieve vacation state only; quota, moderation, policy, custom, and large-audience Exchange tips are not synthesized.
- `GetReminders` returns computed canonical calendar/task/message reminder views.
- `PerformReminderAction` dismisses canonical calendar/task reminders through `reminder_occurrence_dismissals` and snoozes them by updating canonical reminder time/reset state; it does not create EWS reminder rows.
- `GetRooms` projects tenant-visible canonical room and equipment directory accounts only; hidden accounts, person accounts, and other tenants are excluded.
- `GetRoomLists` returns LPE's computed tenant room/resource list when visible room or equipment accounts exist. Arbitrary Exchange room-list membership is not implemented.
- `DeleteItem` maps Exchange delete types onto canonical hard delete or Trash move behavior.

The schema is adequate for the currently implemented bounded EWS surface. It also contains canonical data models for several high-value gaps that are still not EWS-wired, including retention tags, eDiscovery, bulk transfer, mail apps, and unified messaging. Those are storage-ready but not SOAP-compatible until dispatcher handlers, API methods, response shapes, and client-visible tests exist.

Remaining full-parity limits are primarily behavior and protocol semantics, not raw SQL table absence:

- Exchange opaque id conversion across all object families is not implemented.
- Exchange managed/archive/search folder behavior is not fully modeled through EWS.
- Push/streaming subscription affinity is bounded to canonical replay, not Exchange long-lived subscription semantics.
- Room lists are computed from tenant room/resource accounts, not stored as arbitrary Exchange distribution-room-list objects.
- Sharing, mail app, retention, eDiscovery, bulk transfer, UM, and UCS operation handlers are not wired even where canonical schema exists.

## Automated Compatibility Gate

`crates/lpe-exchange/src/tests/ews.rs` now contains `ews_catalog_gate_covers_documented_operations_and_unsupported_gaps`, an automated EWS compatibility gate derived from `docs/audits/ews-parity-matrix-2026-05-30.md`.

The gate:

- parses the documented operation names from the parity matrix;
- requires each operation to appear in the EWS test coverage manifest exactly once;
- verifies every implemented/partial operation points at an existing SOAP behavior test in the EWS test module;
- sends SOAP requests for every unsupported catalog operation and requires a parseable operation-shaped `ErrorInvalidOperation` response that documents the unsupported behavior;
- prints operation-name, behavioral, and unsupported coverage when run with `--nocapture`.

## Conclusions

1. The EWS audit result has changed materially: automated operation-name test coverage is now complete across the documented catalog, while implemented behavioral operation coverage is 48 of 96 operation names.
2. The newly wired operations route through canonical LPE APIs/storage or stateless compatibility responses. I did not find EWS-local Sent, Outbox, reminder, or subscription state competing with canonical state.
3. SQL is sufficient for the bounded implemented EWS behavior and now includes canonical state for several remaining parity gaps.
4. `LPE` is still far from 100% Microsoft EWS compatibility because most remaining work is full SOAP behavior, Exchange response-shape fidelity, operation semantics, and interoperability testing rather than table creation alone.

## Highest-Value Remaining EWS Gaps

Priority for practical Outlook/native-client compatibility:

1. Harden `SendItem` only where real clients require additional Exchange disposition options, while preserving canonical submission.
2. Widen `UpdateInboxRules` only for additional shapes that map safely to canonical Sieve behavior.
3. Widen reminder actions only when they map to canonical reminder fields.
4. Add explicit canonical room-list membership only if clients require more than LPE's computed tenant room/resource list.
5. Harden notification long-poll/streaming affinity only if Outlook/EWS clients require more than bounded canonical replay.
6. Widen `CreateUserConfiguration`, `GetUserConfiguration`, `UpdateUserConfiguration`, and `DeleteUserConfiguration` only if real clients require additional Exchange payload shapes beyond the bounded canonical `account_client_configurations` mapping.
7. Add `ConvertId` only after the canonical opaque-id strategy is settled for every EWS object family.
