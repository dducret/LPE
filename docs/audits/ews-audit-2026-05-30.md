# EWS Audit - 2026-05-30

## Scope

This audit compares `LPE` against Microsoft's EWS operation catalog at:

- <https://learn.microsoft.com/en-us/exchange/client-developer/web-service-reference/ews-operations-in-exchange>
- <https://learn.microsoft.com/en-us/exchange/client-developer/web-service-reference/ews-xml-elements-in-exchange>

This is a static repository audit plus local test verification. It is not a live Exchange or Outlook interoperability run.

## Summary

`LPE` does not implement complete Microsoft Exchange Web Services. It implements a bounded EWS compatibility adapter for canonical mail, contacts, calendar, tasks, attachments, OOF/vacation, availability, and pull notifications.

The EWS dispatcher is explicit in `crates/lpe-exchange/src/service.rs`: supported operations call concrete handlers; a small set of known out-of-scope operations returns explicit EWS error responses; every other recognized operation name falls through to a generic unsupported EWS response.

Verification run:

- `cargo test -p lpe-exchange`: passed, 670 tests.
- `cargo test -p lpe-storage --test runtime_schema_drift`: passed, 1 test.

## Implemented EWS Operations

| Operation | LPE status | API/storage connection | Notes and visible differences |
| --- | --- | --- | --- |
| `SyncFolderHierarchy` | Implemented | `fetch_jmap_mailboxes`, accessible contact/calendar/task collections | Projects canonical folders and collaboration collections; not full Exchange hierarchy including public/archive/voice/search folders. |
| `FindFolder` | Implemented | Same as above | Lists canonical mail folders plus contact/calendar/task folders. |
| `GetFolder` | Implemented | Same as above | Supports selected distinguished folders and canonical mailbox folder ids. Unsupported distinguished ids return folder errors. |
| `FindItem` | Implemented | Mail via JMAP storage; contacts/events/tasks via collaboration storage | Bounded to mail, contacts, calendar, tasks. Defaults to contacts when no folder kind is detected. |
| `GetItem` | Implemented | Same as above, plus attachment reads for MIME content | Bounded item ids: `message:`, `contact:`, `event:`, `task:`. Unsupported ids return `ErrorItemNotFound`. |
| `SyncFolderItems` | Implemented | Canonical current-state queries plus synthetic sync-state tokens | One-way sync over current visible state; not a complete Exchange sync engine. |
| `CreateItem` | Implemented | Contacts/events/tasks canonical upserts; mail draft/import/submission APIs | Supports contacts, calendar items, tasks, mail drafts, custom-folder import, `SendOnly`, and `SendAndSaveCopy`. |
| `UpdateItem` | Implemented | Contact/event/task update APIs; message flag update APIs | Message updates support only read and flag state. |
| `DeleteItem` | Implemented | Canonical contact/event/task deletes; mail delete/move-to-trash APIs | `HardDelete`, `SoftDelete`, and `MoveToDeletedItems` parse, but mail soft-delete maps to canonical Trash move when possible. |
| `MoveItem` | Implemented | Canonical message move API | Supports canonical message ids only. |
| `CopyItem` | Implemented | Canonical message copy API | Supports canonical message ids only. |
| `CreateFolder` | Implemented | Canonical JMAP mailbox create API | Creates a subscribed custom mail folder. Parent folder handling is bounded; no full Exchange folder-class creation. |
| `DeleteFolder` | Implemented | Canonical JMAP mailbox destroy API | Supports canonical mailbox folder ids. Protected/system behavior is enforced by storage. |
| `GetAttachment` | Implemented | Canonical message/calendar attachment content APIs | Supports LPE attachment references, returns file attachment content. |
| `CreateAttachment` | Implemented | Magika validation plus canonical message attachment API | Supports `FileAttachment` on one canonical message parent. `ItemAttachment` is explicitly unsupported. |
| `DeleteAttachment` | Implemented | Canonical message attachment delete API | Supports LPE attachment references. |
| `ResolveNames` | Implemented | Authenticated account, tenant directory accounts, accessible contacts | No full GAL or directory template behavior. |
| `GetUserAvailability` | Implemented | Canonical calendar events | Only authenticated mailbox free/busy is supported. |
| `GetServerTimeZones` | Implemented | Static response | Returns minimal compatibility definitions, not full Exchange time-zone corpus. |
| `GetUserOofSettings` | Implemented | Active canonical Sieve script | OOF is projected from canonical vacation Sieve, not an Exchange OOF table. |
| `SetUserOofSettings` | Implemented | Canonical Sieve script create/activate/deactivate | `Enabled` and `Scheduled` map to vacation Sieve; disabling clears active script. |
| `Subscribe` | Implemented | In-process pull subscription registry and event journal | Pull subscriptions only; push and streaming subscriptions are out of scope. |
| `GetEvents` | Implemented | In-process event queue with fallback canonical mailbox projections | Not a durable Exchange notification queue. |
| `Unsubscribe` | Implemented | Parses subscription id and returns success | Does not persist subscription state in SQL. |

## Workarounds and Compatibility Behavior in LPE

Visible workarounds or bounded compatibility choices:

- `/EWS/Exchange.asmx` and `/ews/exchange.asmx` both route to the same EWS handler.
- EWS authentication accepts the shared mailbox account authentication path and returns a Basic challenge for EWS.
- Operation detection is XML-name based rather than schema/WSDL validation.
- Unsupported operations return SOAP/EWS-shaped error payloads instead of raw HTTP failures.
- `SyncFolderItems` uses client-visible synthetic sync-state strings derived from current canonical state instead of durable Exchange sync cursors.
- Pull notification subscriptions and event journals are in process, have TTLs, and are not stored in SQL.
- `GetEvents` can emit compatibility created/new-mail/status events from current mailbox state when no durable event queue exists.
- `GetServerTimeZones` returns a small static compatibility set.
- OOF is translated to and from canonical Sieve vacation scripts.
- `CreateAttachment` validates uploaded bytes with Magika before canonical attachment storage and supports only file attachments.
- EWS item ids are LPE-prefixed canonical ids such as `message:<uuid>`, `contact:<uuid>`, `event:<uuid>`, and `task:<uuid>`, not Exchange opaque ids.
- Calendar availability is scoped to the authenticated mailbox.
- Mail `CreateItem` with send disposition goes through canonical submission and authoritative `Sent`; no EWS outbox or client SMTP path exists.
- Mail delete maps Exchange delete types to canonical hard delete or Trash move behavior.
- Contacts, events, tasks, and mail are projected from canonical LPE tables rather than Exchange-specific item stores.

## SQL Data Sufficiency

The SQL schema contains the durable data needed for LPE's bounded EWS implementation:

| EWS need | SQL support |
| --- | --- |
| Authentication and tenant ownership | `tenants`, `domains`, `accounts`, `account_credentials`, `account_sessions`, OIDC/app-password tables |
| Mail folders and messages | `mailboxes`, `mailbox_messages`, `messages`, `message_headers`, `message_recipients`, `protected_bcc_recipients`, `message_bodies`, `mail_search_documents` |
| Attachments | `blobs`, `blob_placements`, `mime_parts`, `attachments`, `attachment_extraction_jobs`, `attachment_texts`, `calendar_event_attachments` |
| Contacts | `contact_books`, `contacts`, `contact_book_grants` |
| Calendar and free/busy | `calendars`, `calendar_events`, `calendar_grants` |
| Tasks | `task_lists`, `tasks`, `task_list_grants` |
| OOF | `sieve_scripts`, `sieve_vacation_responses` |
| Sending and Sent visibility | `account_identities`, `sender_rights`, `submission_queue`, `submission_recipients`, `submission_events`, canonical message/mailbox tables |
| Cross-protocol sync and deletes | `account_sync_state`, `mail_change_log`, `tombstones`, `recoverable_items` |
| Auditability | `audit_events` |

The schema is not sufficient for full Microsoft EWS parity because there is no durable model for Exchange eDiscovery configuration, non-indexable item reports, message tracking reports, mail apps, Unified Messaging, Unified Contact Store / IM groups, Exchange user-configuration blobs, retention policy tags, durable EWS notification subscriptions, full EWS delegate objects, public folders, archive folders, managed folders, room lists, or EWS bulk import/export streams.

## Conclusions

1. All EWS operations that LPE claims in documentation are implemented in the dispatcher and connected to canonical storage APIs.
2. LPE does not implement all Microsoft EWS operations. The visible difference is intentional bounded compatibility, not a hidden full-Exchange implementation.
3. The SQL database contains the data required for the bounded LPE EWS surface.
4. The SQL database does not contain all data needed for complete Microsoft EWS parity.
5. The largest operational gaps for Exchange-style clients are full `SendItem`, `GetInboxRules` / `UpdateInboxRules`, reminders, delegate management, room/resource discovery, durable notifications, user configuration, and bulk upload/export.
