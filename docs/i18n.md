# Internationalized Mailbox Support Plan

Accessed standards date: 2026-05-13.

## Executive summary

LPE is not ready to enable internationalized mailbox support. The canonical
mailbox model is usable, but mailbox names are currently handled as plain
strings with ASCII-oriented comparison, rendering, wildcard matching, and
validation. The shared mailbox-name policy is now defined in
`docs/architecture/internationalized-mailbox-names.md`. The implementation
direction is IMAP4rev2 with UTF-8 mailbox names, `UTF8=ACCEPT` in the first
internationalized mailbox release, persistent subscription state, and JMAP Mail
over canonical LPE mailbox state with `parentId` hierarchy support.

The implementation should be staged: first use the documented Unicode and
mailbox-name policy, then add shared mailbox-name parsing and validation, then
integrate IMAP and JMAP behavior, then add interoperability and security tests.

## Current implementation-readiness assessment

LPE should not enable internationalized mailbox support yet.

Relevant current behavior:

- IMAP capabilities are declared in `crates/lpe-imap/src/service.rs`.
- IMAP mailbox parsing is token-based in `crates/lpe-imap/src/parse.rs` and
  does not use a mailbox-name abstraction.
- IMAP create and rename normalize mailbox paths by trimming and splitting on
  `/` in `crates/lpe-imap/src/mailboxes.rs`.
- IMAP list, status, and selected-mailbox rendering quote raw strings in
  `crates/lpe-imap/src/render.rs`.
- IMAP matching uses ASCII-oriented comparison and byte-oriented wildcard
  behavior.
- JMAP Mailbox objects are exposed through `crates/lpe-jmap/src/mailboxes.rs`.
  Names are returned directly, `isSubscribed` is currently always true,
  `parentId` is not exposed, and child creation rights are false. These are
  gaps: internationalized mailbox support requires persisted subscriptions and
  immediate `parentId` hierarchy support.
- Storage duplicate checks use `lower(display_name)` in
  `crates/lpe-storage/src/protocols.rs`.
- Account email normalization has EAI and IDNA-related behavior in
  `crates/lpe-storage/src/util.rs`, but mailbox names do not share a Unicode
  policy.

## Internationalized mailbox semantic gaps

Must-fix before enabling:

- Implement the documented canonical mailbox-name normalization policy.
- Add shared mailbox display-name, path, and comparison-key abstractions.
- Make IMAP mailbox parsing and serialization UTF-8-aware under IMAP4rev2.
- Make hierarchy delimiter parsing segment-aware.
- Replace ASCII-only comparison and wildcard matching with Unicode-aware
  mailbox-name behavior.
- Reject invalid Unicode, controls, unsafe invisible characters, and invalid
  hierarchy paths.
- Detect canonical-equivalent sibling collisions, such as `Cafe\u0301` and
  `Café`.
- Add tests for non-ASCII names, emoji, RTL text, combining marks, malformed
  input, and spoofing-sensitive characters.

Settled product decisions:

- Include IMAP `UTF8=ACCEPT` in the first internationalized mailbox release.
- Support JMAP `parentId` immediately.
- Persist real subscription state now.
- Enforce strict mixed-script and confusable-name validation based on UTS #39.
- Store canonical backend mailbox names and roles in standard, predictable
  server form, such as `INBOX`, `Sent`, and `Trash`.
- Keep localized folder labels strictly in client UI presentation.
- Provide server-side special-use metadata to guide client presentation layers.
- Map special-use metadata directly to JMAP `role` properties.
- Use the `unicode-normalization` crate for NFC canonical processing.
- Use the `unicode-security` crate for mixed-script and confusable detection.

Should-fix during early implementation:

- Align account-address EAI handling with mailbox ownership checks.

Optional improvements:

- Add UI display safeguards for bidi and mixed-script names.
- Add client-lab transcripts for Outlook, Thunderbird, Apple Mail, and mobile
  clients with internationalized folders.

## IMAP interoperability considerations

IMAP mailbox names should be parsed and emitted as UTF-8 mailbox names under
IMAP4rev2. `UTF8=ACCEPT` is included in the first internationalized mailbox
release and must be advertised only when the parser, serializer, validation, and
tests match the implemented behavior. RFC 9051 section 5.1 governs mailbox
naming and hierarchy behavior.
`LIST`, `LSUB`, `STATUS`, `SELECT`, `EXAMINE`, `CREATE`, `RENAME`, `DELETE`,
`SUBSCRIBE`, and `UNSUBSCRIBE` must all use the same decoded mailbox-name model.

The hierarchy delimiter should remain `/` unless LPE explicitly documents a
different delimiter. A delimiter is protocol syntax, not a valid character
inside a mailbox segment. LIST wildcard behavior must be implemented over
decoded hierarchy segments: `%` matches one hierarchy level and `*` matches
recursively as defined by IMAP LIST semantics.

Special-use flags such as `\Sent`, `\Drafts`, and `\Trash` should be emitted as
attributes, not inferred from localized display names. `\Archive`, `\Junk`, and
other supported special-use attributes should follow the same role-based model.
This follows RFC 6154.

## JMAP interoperability considerations

JMAP mailbox names are JSON strings and should be returned as user-visible
Unicode strings. RFC 8621 section 2 defines the Mailbox object and its `name`,
`parentId`, `role`, `sortOrder`, `isSubscribed`, and rights fields. LPE will
support `parentId` immediately for internationalized mailbox work and will
persist subscription state so `isSubscribed` reflects canonical account state.
JMAP does not define LPE's Unicode normalization or collision policy, so LPE
must document and enforce one consistently.

JMAP `role` should remain canonical and language-neutral. Localized mailbox
labels should not change role identity. JMAP creation and update requests that
set hierarchy fields must validate parent ownership, prevent cycles, and enforce
canonical sibling collision checks.

Canonical backend names such as `INBOX`, `Sent`, and `Trash` remain stable on
the server. Client applications are responsible for presenting localized labels,
using JMAP `role` values and IMAP special-use attributes as metadata.

## Proposed implementation architecture

Add a shared mailbox-name module in `lpe-domain` or `lpe-storage`:

- `MailboxDisplayName`: validated user-visible NFC string.
- `MailboxSegment`: one validated hierarchy segment.
- `MailboxPath`: ordered decoded hierarchy segments.
- `MailboxCanonicalKey`: comparison key derived from NFC plus the chosen
  Unicode case-folding and security policy.
- `MailboxNamePolicy`: validation, normalization, reserved-name handling, and
  collision detection.
- `ImapMailboxName`: IMAP parser and serializer over UTF-8 mailbox names.
- `JmapMailboxName`: JMAP validation adapter over the same policy.

Canonical backend mailbox names and role metadata remain server-side identity
and interoperability data. Localized display strings are UI concerns, not stored
backend mailbox identities.

Protocol adapters should use this module and avoid local mailbox-name rules.
Storage remains the canonical state owner.

## Normalization and comparison policy

Recommended policy:

- Normalize stored display names to Unicode NFC, based on UAX #15 and RFC 5198.
- Preserve user-visible spelling after NFC normalization.
- Use a generated canonical key for duplicate detection.
- Apply Unicode case folding for sibling-name collision checks.
- Treat `INBOX` as a reserved IMAP name and resolve it through the canonical
  inbox role.
- Reject canonical-equivalent sibling duplicates.
- Do not rely on PostgreSQL `lower(...)` as the final Unicode comparison
  policy.

This comparison policy is an engineering inference because RFC 8621 does not
prescribe Unicode case folding for mailbox names.

Implementation crate:

- Use `unicode-normalization` for NFC processing.
- The selected crate is acceptable under `LICENSE.md` because it is published
  under MIT/Apache-2.0-compatible licensing.

## Validation and security policy

Reject:

- empty mailbox names,
- empty hierarchy segments,
- names exceeding documented character and UTF-8 byte limits,
- `/` inside a segment while `/` is the hierarchy delimiter,
- NUL,
- C0 and C1 controls,
- CR and LF,
- default-ignorable invisible characters unless explicitly allowlisted,
- bidi override and isolate controls unless a safe profile is documented,
- names that canonicalize to reserved special-use or `INBOX` identities without
  being the corresponding canonical role,
- mixed-script confusables,
- names visually close to special-use mailboxes,
- whole-script confusables identified by UTS #39.

Filesystem safety:

- Mailbox display names must never be used directly as filesystem paths.
- Any future path mapping must use opaque mailbox IDs.

Implementation crate:

- Use `unicode-security` for strict mixed-script and confusable detection based
  on UTS #39.
- The selected crate is acceptable under `LICENSE.md` because it is published
  under MIT/Apache-2.0-compatible licensing.

## Risk-ranked issue list

Must-fix before enabling:

1. Shared Unicode mailbox-name policy is documented but not implemented in code.
2. IMAP mailbox parser and serializer are not UTF-8 mailbox-name aware.
3. IMAP LIST wildcard matching is byte/ASCII oriented.
4. Duplicate checks are not canonical-equivalence aware.
5. Invalid Unicode and unsafe control characters are not rejected through one
   shared policy.
6. JMAP Mailbox creation and update do not use shared mailbox-name validation.
7. Tests do not cover Unicode mailbox names.
8. IMAP `UTF8=ACCEPT` is not implemented or advertised.
9. Subscription state is not persisted.

Should-fix:

1. Special-use aliases are English-display-name driven.
2. EAI account identity and mailbox ownership checks are not documented together.

Optional:

1. Add localized UI labels while keeping canonical roles stable.
