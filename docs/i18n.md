# Internationalized Mailbox Support Plan

Accessed standards date: 2026-05-13. Last implementation audit:
2026-05-14.

## Executive summary

LPE now has the planned internationalized mailbox implementation in place for
the audited IMAP, JMAP, shared domain, and storage-contract paths. The shared
mailbox-name policy is documented in
`docs/architecture/internationalized-mailbox-names.md`, and the code now has a
shared `lpe-domain::mailbox_name` module, NFC normalization through
`unicode-normalization`, UTS #39 skeleton and mixed-script checks through
`unicode-security`, IMAP UTF-8 mailbox-name tests, canonical hierarchy storage,
JMAP `parentId`, JMAP `mayCreateChild` rights, and persisted subscription state
shared by IMAP and JMAP.

The previous release blockers from this plan are resolved in the audited code:
IMAP nested mailbox paths persist as canonical hierarchy rows using
`parent_mailbox_id`; canonical system mailbox names use `INBOX`, `Sent`,
`Drafts`, `Trash`, `Junk`, and `Archive`; the legacy IMAP capability is no
longer advertised; malformed IMAP command literals are rejected as invalid
UTF-8; and JMAP child-creation rights now match implemented `parentId` create
behavior. Remaining work is release validation: real-client interoperability
transcripts, continued Unicode security data tracking, and EAI/account-identity
documentation alignment.

## Current implementation-readiness assessment

LPE is implementation-complete for the audited internationalized mailbox scope.
Production enablement should still wait for release validation with target
clients and deployment-specific database/runtime checks.

Implemented behavior:

- `crates/lpe-domain/src/mailbox_name.rs` defines shared
  `MailboxDisplayName`, `MailboxSegment`, `MailboxPath`,
  `MailboxCanonicalKey`, and `MailboxNamePolicy` types.
- The shared mailbox-name module normalizes accepted names to NFC with
  `unicode-normalization`, rejects controls, unsafe invisible characters,
  delimiter-in-segment names, empty segments, and reserved-name spoofing, and
  generates canonical equality and skeleton keys.
- IMAP mailbox parsing in `crates/lpe-imap/src/parse.rs` now uses
  `MailboxPath` for command mailbox operands.
- IMAP rendering and matching in `crates/lpe-imap/src/render.rs` and
  `crates/lpe-imap/src/mailboxes.rs` use shared mailbox-name matching and
  Unicode-aware LIST wildcard tests over normalized text.
- IMAP advertises and implements `ENABLE UTF8=ACCEPT`, and tests cover UTF-8
  mailbox-name response quoting before and after enabling it.
- IMAP capability advertisement now contains the documented implemented
  capability set and does not advertise the removed legacy protocol target.
- IMAP command literals for mailbox commands are decoded as strict UTF-8; APPEND
  message literals remain byte-oriented.
- JMAP Mailbox create and update paths in `crates/lpe-jmap/src/mailboxes.rs`
  validate names with `MailboxSegment`, expose `parentId`, and persist
  `isSubscribed`.
- JMAP `myRights.mayCreateChild` reflects mailbox account write access.
- Storage paths in `crates/lpe-storage/src/protocols.rs` use shared canonical
  collision checks before mailbox create and rename operations.
- IMAP `CREATE` and `RENAME` for nested paths such as `Projects/Alpha` now
  persist each hierarchy segment as one mailbox row and update
  `parent_mailbox_id` instead of storing `/` inside `display_name`.
- `mailbox_subscriptions` now exists in `crates/lpe-storage/sql/schema.sql` and
  is used by IMAP `SUBSCRIBE` / `UNSUBSCRIBE` / `LSUB` and JMAP
  `isSubscribed`.
- Canonical system mailbox names are `INBOX`, `Drafts`, `Sent`, `Trash`,
  `Junk`, and `Archive`, with compatibility aliases resolved by role.
- Tests cover NFC, `Café` versus `Cafe\u0301`, emoji, CJK, RTL names, unsafe
  invisible and bidi controls, mixed-script examples, JMAP `parentId`, IMAP
  `UTF8=ACCEPT`, IMAP `LIST` hierarchy wildcards, and subscription state.

Release validation still required:

- `unicode-security` is present in `Cargo.toml` / `Cargo.lock`, and the shared
  mailbox-name policy uses its UTS #39 skeleton and mixed-script detection
  helpers instead of a local limited confusable table. The only retained local
  correction is the current UTS #39 U+04CF skeleton mapping to Latin `l`,
  because `unicode-security` 0.1.2 is generated from Unicode 16.0 data.
- Database generated `normalized_display_name` columns still use
  `lower(display_name)`. Runtime storage APIs enforce the shared NFC,
  canonical-key, reserved-name, and confusable policy; the database generated
  column is only a compatibility guard and is not the Unicode canonical
  identity policy.
- Real-client interoperability transcripts are still required before broad
  release enablement.

## IMAP interoperability considerations

IMAP mailbox names should be parsed and emitted as UTF-8 mailbox names under
IMAP4rev2. `UTF8=ACCEPT` is included in the first internationalized mailbox
release and must be advertised only when the parser, serializer, validation, and
tests match the implemented behavior. RFC 9051 section 5.1 governs mailbox
naming and hierarchy behavior.
`LIST`, `LSUB`, `STATUS`, `SELECT`, `EXAMINE`, `CREATE`, `RENAME`, `DELETE`,
`SUBSCRIBE`, and `UNSUBSCRIBE` must all use the same decoded mailbox-name model.

The hierarchy delimiter should remain `/` unless LPE explicitly documents a
different delimiter. A delimiter is protocol syntax, not a valid character
inside a mailbox segment. LIST wildcard behavior must be implemented over
decoded hierarchy segments: `%` matches one hierarchy level and `*` matches
recursively as defined by IMAP LIST semantics.

Special-use flags such as `\Sent`, `\Drafts`, and `\Trash` should be emitted as
attributes, not inferred from localized display names. `\Archive`, `\Junk`, and
other supported special-use attributes should follow the same role-based model.
This follows RFC 6154.

## JMAP interoperability considerations

JMAP mailbox names are JSON strings and should be returned as user-visible
Unicode strings. RFC 8621 section 2 defines the Mailbox object and its `name`,
`parentId`, `role`, `sortOrder`, `isSubscribed`, and rights fields. LPE will
support `parentId` immediately for internationalized mailbox work and will
persist subscription state so `isSubscribed` reflects canonical account state.
JMAP does not define LPE's Unicode normalization or collision policy, so LPE
must document and enforce one consistently.

JMAP `role` should remain canonical and language-neutral. Localized mailbox
labels should not change role identity. JMAP creation and update requests that
set hierarchy fields must validate parent ownership, prevent cycles, and enforce
canonical sibling collision checks.

Canonical backend names such as `INBOX`, `Sent`, and `Trash` remain stable on
the server. Client applications are responsible for presenting localized labels,
using JMAP `role` values and IMAP special-use attributes as metadata.

## Proposed implementation architecture

Add a shared mailbox-name module in `lpe-domain` or `lpe-storage`:

- `MailboxDisplayName`: validated user-visible NFC string.
- `MailboxSegment`: one validated hierarchy segment.
- `MailboxPath`: ordered decoded hierarchy segments.
- `MailboxCanonicalKey`: comparison key derived from NFC plus the chosen
  Unicode case-folding and security policy.
- `MailboxNamePolicy`: validation, normalization, reserved-name handling, and
  collision detection.
- `ImapMailboxName`: IMAP parser and serializer over UTF-8 mailbox names.
- `JmapMailboxName`: JMAP validation adapter over the same policy.

Canonical backend mailbox names and role metadata remain server-side identity
and interoperability data. Localized display strings are UI concerns, not stored
backend mailbox identities.

Protocol adapters should use this module and avoid local mailbox-name rules.
Storage remains the canonical state owner.

## Normalization and comparison policy

Recommended policy:

- Normalize stored display names to Unicode NFC, based on UAX #15 and RFC 5198.
- Preserve user-visible spelling after NFC normalization.
- Use a generated canonical key for duplicate detection.
- Apply Unicode case folding for sibling-name collision checks.
- Treat `INBOX` as a reserved IMAP name and resolve it through the canonical
  inbox role.
- Reject canonical-equivalent sibling duplicates.
- Do not rely on PostgreSQL `lower(...)` as the final Unicode comparison
  policy.

This comparison policy is an engineering inference because RFC 8621 does not
prescribe Unicode case folding for mailbox names.

Implementation crate:

- Use `unicode-normalization` for NFC processing.
- The selected crate is acceptable under `LICENSE.md` because it is published
  under MIT/Apache-2.0-compatible licensing.

## Validation and security policy

Reject:

- empty mailbox names,
- empty hierarchy segments,
- names exceeding documented character and UTF-8 byte limits,
- `/` inside a segment while `/` is the hierarchy delimiter,
- NUL,
- C0 and C1 controls,
- CR and LF,
- default-ignorable invisible characters unless explicitly allowlisted,
- bidi override and isolate controls unless a safe profile is documented,
- names that canonicalize to reserved special-use or `INBOX` identities without
  being the corresponding canonical role,
- mixed-script confusables,
- names visually close to special-use mailboxes,
- whole-script confusables identified by UTS #39.

Filesystem safety:

- Mailbox display names must never be used directly as filesystem paths.
- Any future path mapping must use opaque mailbox IDs.

Implementation crate:

- Use `unicode-security` for strict mixed-script and confusable detection based
  on UTS #39.
- The selected crate is acceptable under `LICENSE.md` because it is published
  under MIT/Apache-2.0-compatible licensing.
- `unicode-security` 0.1.2 is generated from Unicode 16.0 data. Until the crate
  ships current UTS #39 data, LPE retains a narrow correction for U+04CF
  CYRILLIC SMALL LETTER PALOCHKA so the security skeleton maps it to Latin
  `l`, matching current UTS #39 data.

## Open questions and assumptions

Assumptions:

- IMAP mailbox support targets IMAP4rev2 with UTF-8 mailbox names.
- IMAP `UTF8=ACCEPT` is included in the first release.
- `/` remains the hierarchy delimiter.
- JMAP mailbox names are user-visible Unicode JSON strings.
- JMAP `parentId` hierarchy is supported immediately.
- Subscription state is persisted now.
- Mixed-script and confusable mailbox names are rejected under strict
  validation.
- SQL schema design is out of scope for the planning phase.
- MAPI over HTTP is out of scope.
- Canonical backend names remain standard and predictable; localized labels are
  not backend identity.
- `unicode-normalization` is the selected NFC crate.
- `unicode-security` is the selected strict mixed-script/confusable detection
  crate.

Open questions:

- None currently.
