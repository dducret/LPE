# Outlook Exchange Parity Roadmap

This roadmap consolidates the current Outlook and Exchange compatibility gaps
from the active MAPI, EWS, and interoperability architecture contracts.

It is documentation-only. It does not change endpoint publication, runtime
behavior, schema, or release readiness.

## Scope Rules

- `LPE` remains the canonical store for mailbox, collaboration, submission,
  rights, search, recovery, and user-visible state.
- Outlook and Exchange compatibility metadata may be durable only when it
  supports protocol identity, profile reuse, associated configuration, or sync
  replay. It must not become mailbox truth.
- Public MAPI autodiscover is enabled for the 0.5.0 baseline when
  `LPE_AUTOCONFIG_MAPI_ENABLED` is true. Local harness, Microsoft RCA, and real
  Outlook cached-mode runs remain release-quality evidence rather than a
  second runtime gate.
- A gap marked as not blocking public MAPI autodiscover can still block broader
  Exchange parity, EWS parity, native-client parity, or a later production
  hardening gate.

Mutations must go through contact/account APIs, not NSPI-local state. | NSPI bootstrap tests, ambiguous-name ranking tests, distribution-list expansion tests, tenant visibility tests, and real Outlook recipient-resolution evidence. | Yes for bind/resolve/profile bootstrap; mutation/template parity is broader Outlook functionality debt. |

## Priority Interpretation

- `Yes` in the final column means the gap can block public MAPI autodiscover
  publication for the first Outlook desktop Exchange-account path.
- `No` means the gap is still tracked, but it belongs to broader EWS, Exchange,
  native-client, production-hardening, or feature-family parity.
- Some rows are conditional because the source docs intentionally separate the
  first Outlook 2016 / 2019 cached-mode lab gate from later production or
  legacy compatibility gates.

## Verification Expectations

Before a gap can be closed, update the relevant source matrix or architecture
document and add the evidence named in its row. At minimum:

- MAPI closure needs local `lpe-exchange` protocol tests plus real Outlook
  profile evidence when it affects profile creation, sync, send, reconnect, or
  autodiscover.
- EWS closure needs the EWS catalog gate to name SOAP behavior tests or an
  explicit unsupported reason for every affected Microsoft operation.
- Canonical-state closure needs cross-protocol tests proving JMAP, IMAP where
  applicable, EWS, and MAPI observe the same user-visible state.
- Any behavior that cannot map to current canonical state must first add or
  document a canonical model. Opaque Exchange blobs must not be accepted as
  active behavior by default.
