# MAPI over HTTP Implementation Plan

## Current State/Functionality Overview

This document is the Microsoft specification-to-`LPE` implementation reference for the guarded MAPI over HTTP compatibility track.

The reference is documentation-only. It does not change runtime behavior. `LPE` remains the canonical store for mailboxes, contacts, calendars, tasks, rights, submission, and user-visible state, while MAPI over HTTP stays an authenticated Outlook desktop compatibility surface over that canonical state.

The initial ticket sequence has been completed and removed from this document. The remaining matrices describe implemented coverage, release gates, and deferred surfaces.

The Microsoft protocol specifications were checked on 2026-05-17 for `MS-OXCFXICS` `RopSynchronizationConfigure`, `hierarchySync`, `contentsSync`, `messageChange`, read-state changes, deletion state, final ICS state, `IDSET` / `CNSET` syntax, and FastTransfer property lexical encoding. The cited specifications define wire contracts and data structures, but they do not define Outlook's exact tolerance for partial Exchange compatibility. Real Outlook 2016 and Outlook 2019 profile creation, cached-mode synchronization, reconnect, and RCA evidence remain required release gates.

MAPI over HTTP is the first Outlook desktop Exchange-account path for this work. Outlook Anywhere / RPC over HTTP is a legacy compatibility shim for later `EXPR` publication and must not drive the initial MAPI over HTTP implementation order.

Single-node sticky MAPI sessions are acceptable for the first Outlook 2016 / 2019 lab gate. Cross-process session replay, durable session migration, and load-balanced session failover remain production hardening.

Readiness terminology is intentionally split:

- Local harness pass means `crates/lpe-exchange` tests and project-owned live scripts, including `tools/rca_outlook_connectivity_check.py --outlook-rca-readiness`, pass against a configured deployment. This proves LPE's scripted compatibility path, not public Outlook readiness by itself.
- RCA pass means Microsoft Remote Connectivity Analyzer Outlook Connectivity passes from the Internet against the same public host and account that will be advertised.
- Real Outlook profile pass means Outlook 2016 and Outlook 2019 each create an Exchange account profile, complete cached-mode synchronization, close and reopen without a full-cache wipe, resolve NSPI entries, send through canonical submission, and show the authoritative `Sent` copy.

Current MAPI status: local harness and server-side hierarchy-sync traces have passed through the stages recorded below, but the full publication gate remains pending until Microsoft RCA evidence and real Outlook 2016 / 2019 cached-mode profile evidence are both recorded.

Successful EMSMDB `Execute` responses refresh both the `MapiContext` session cookie and the matching `MapiSequence` cookie so request replay and later session-bound requests can continue with a complete single-node in-process session context.

Reserved or otherwise unsupported ROPs are terminal within the current ROP request buffer: the server returns one parseable unsupported ROP error and does not execute later ROP bytes from that batch.

Typed protocol enum boundaries are strict. Unknown `RopId`, MAPI property type, restriction type, synchronization type, and FastTransfer marker values remain outside the typed enum variants, are logged with their raw numeric value, and then follow the documented unsupported or parseable error path. They must not be coerced into a nearest supported value, must not panic, and must not create canonical mailbox, contact, calendar, task, rights, submission, or sync-state side effects.

`RopLogon` creates a store handle only for private mailbox logons. Public-folder logons, identified by the request `Private` flag not being set, remain deferred with a parseable unsupported ROP response and no protocol-local public-folder store state.

## Outlook Special Folders Execution Plan

This plan covers Outlook `Journal`, `Notes`, `Tasks`, and `Reminders` support across canonical storage, JMAP, IMAP, MAPI over HTTP, and the client web UI. It is ordered for correctness rather than compatibility-preserving increments because `LPE` is not yet in production.

Key readiness finding: `Tasks` already have canonical storage, JMAP methods, DAV projection, web workspace data, and MAPI task-folder projection. `Journal` and `Notes` now have canonical account-owned item tables and client API endpoints, but MAPI ROP behavior is still deferred. `Reminders` has a MAPI special-folder identifier, but Microsoft defines it as a search folder, so it must not become an ordinary canonical mailbox or protocol-local MAPI store.

### Ordered Implementation

1. Canonical model first.
   - Implemented foundation: canonical `notes` and `journal_entries` tables with account ownership, tenant ownership, title/body or journal-specific fields, timestamps, source metadata, and projection-ready Outlook metadata.
   - Implemented foundation: `calendar_events` and `tasks` carry reminder set/time/dismissal metadata, tasks carry `reminder_reset`, and recurring calendar/task reminders can be expanded for the bounded Reminders projection.
   - Implemented foundation: Reminders stay a computed query over canonical calendar/task/mail reminder metadata. Notes and Journal are not included in reminder results until Microsoft docs or an explicit engineering note justifies expanding the reminder domain.
   - Implemented foundation: Outlook Conversation Action Settings FAI messages are backed by canonical `conversation_actions` rows keyed by the conversation id from bytes 6-21 of `PidTagConversationIndex`. The model persists subject, categories, move folder/store entry IDs, resolved same-store target mailbox id, max-delivery time, last-applied time, version, and processed flags, and writes canonical change rows instead of using a protocol-local FAI store.
   - Implemented foundation: built-in Exchange search folders now have persisted canonical definitions in `search_folders` for Reminders, To-Do, Contacts Search, and Tracked Mail Processing. These definitions store role, display name, result kind, scope JSON, restriction JSON, and excluded mailbox roles, and MAPI store snapshots load them through `ExchangeStore`. Common Views associated contents now project read-only FAI-style definition rows for those persisted definitions, including stable MAPI identities and `PidTagSearchFolderDefinition`/storage/tag metadata. Contacts Search has the first evaluator scaffold: when the canonical built-in definition exists, MAPI contents projection can resolve contact results from canonical contact folders and still applies the caller's table restriction. To-Do now has a bounded evaluator for canonical task objects and follow-up mail backed by persisted canonical mail fields for `PidTagFlagStatus`, `PidTagFollowupIcon`, `PidTagToDoItemFlags`, `PidLidFlagRequest`, `PidLidTaskStartDate`, `PidLidTaskDueDate`, `PidTagFlagCompleteTime`, `PidTagSwappedToDoStore`, and validated `PidTagSwappedToDoData`. Tracked Mail Processing has a bounded evaluator over messages where `PidTagSwappedToDoStore` exists. Reminders now advertises `PidTagRemOnlineEntryId` and direct `OpenFolder` as a computed search-folder projection over canonical calendar, task, and mail reminders. Reminder table rows are links to the underlying appointment, task, or message objects, with `QueryColumnsAll` limited to common identity/class/source/change fields plus reminder set/time/signal properties. `PidTagAdditionalRenEntryIdsEx` advertises the documented Outlook extended special-folder PersistIDs for RSS, Tracked Mail Processing, To-Do, Conversation Action Settings, Suggested Contacts, Contacts Search, IM Contact List, and Quick Contacts through full store folder EntryIDs.

2. JMAP before compatibility adapters.
   - Implemented foundation: JMAP Mail `Mailbox` roles remain standard mail roles; `Journal` and `Notes` are exposed through the documented private LPE capability `https://l-p-e.ch/jmap/outlook`.

   - Implemented foundation: ActiveSync mailbox `Sync` projects flagged or completed canonical mail as the documented `email:Flag` container with `Status`, common `FlagType`, paired Tasks start/due UTC dates when present, and completion times when complete. ActiveSync `Change` commands can set, complete, or clear the same canonical follow-up state.
   - Implemented foundation: the web client exposes an authenticated `PUT /mail/messages/{message_id}/flag` adapter and reading-pane Flag/Clear Flag, Complete/Reopen, Today, Tomorrow, No Date, 1 Hour, Remind Tomorrow, and No Reminder controls that write the canonical mailbox-message follow-up flag state instead of maintaining web-only UI flags. Completion uses the same `complete` follow-up status and completion timestamp observed by MAPI and ActiveSync projections; Today/Tomorrow write canonical follow-up start/due timestamps, No Date clears those timestamps without clearing the flag itself, and reminder controls write canonical reminder set/time metadata that is included in the computed Reminders query and MAPI Reminders search-folder projection.
   - Implemented foundation: Suggested Contacts, Quick Contacts, and IM Contact List are canonical contact-book roles exposed through JMAP Contacts `AddressBook` / `ContactCard`, not through JMAP Mail or IMAP. ## Exchange Special-Folder Baseline

LPE maintains stable MAPI folder identities for the MS-OXOSFLD default special-folder set, but runtime Outlook hierarchy sync currently advertises only the Outlook-safe IPM subtree rows proven to advance the profile lab: the canonical mail and collaboration folders, Suggested Contacts, Sync Issues and its child folders, Junk E-mail, RSS Feeds, Archive, and Conversation History. Quick Contacts, IM Contact List, Contacts Search, Document Libraries, Tracked Mail Processing, To-Do, and Conversation Action Settings keep stable folder identities and direct-open compatibility; the Outlook store property `PidTagAdditionalRenEntryIdsEx` now advertises the documented extended PersistIDs for the implemented direct-open set, but optional/search compatibility folders still are not all emitted as normal hierarchy-sync `folderChange` rows until lab traces prove Outlook accepts them in the initial hierarchy transfer. Root-level compatibility folders include the existing common/view/search/shortcut folders and FreeBusy Data.

JMAP and IMAP expose only folders that are honest canonical mail folders: Inbox, Drafts, Sent, Trash, Junk, Archive, Outbox, RSS Feeds, Conversation History, Sync Issues, Conflicts, Local Failures, and Server Failures. IMAP emits standard `SPECIAL-USE` attributes only for standard roles such as `\Sent`, `\Drafts`, `\Trash`, `\Junk`, and `\Archive`; Outlook-only folders are listed by name without invented IMAP special-use flags. Contact, calendar, task, RSS-object, free/busy, FAI, and search-folder object semantics remain on their canonical protocol surfaces and must not be faked as ordinary IMAP mail.

MAPI over HTTP exposes stable FIDs, source keys, change keys, hierarchy rows, and container classes for the bounded Outlook special-folder set. During initial Outlook bootstrap, `PidTagAdditionalRenEntryIdsEx` advertises full folder EntryIDs for RSS Subscriptions, Tracked Mail Processing, To-Do Search, Conversation Action Settings, Suggested Contacts, Contacts Search, IM Contact List, and Quick Contacts. Those EntryIDs are direct-open compatible through EMSMDB. Search folders are identified with `FOLDER_SEARCH` only where Microsoft documents the folder as a search folder, and their contents are backed by the current canonical search-folder definitions and bounded evaluators. Conversation Action Settings now has canonical FAI read/write semantics backed by `conversation_actions`: Outlook can create, list, open, update, delete, and content-sync associated `IPM.ConversationAction` rows. Same-store move/ignore actions are applied to existing matching non-Sent messages, later MAPI-imported messages that carry the same conversation id in `PidTagConversationIndex`, and inbound-delivered messages once final delivery supplies a matching canonical thread id. Cross-store move requests are preserved on the FAI action and intentionally do not move local mailbox state. Category names are retained on the action and projected onto matching mail messages through `mailbox_messages.keywords`, JMAP `Email.keywords`, and atom-safe IMAP flags. This is a bounded compatibility layer, not a claim that LPE implements every object protocol listed beside those folders.

## Outlook Hierarchy Count Interoperability Experiment

`LPE_MAPI_EXPERIMENT_FORCE_HIERARCHY_COUNT_PROPERTIES` is retired as a mutating MAPI/HTTP experiment. The environment flag can still be logged as requested trace context, but authenticated EMSMDB hierarchy sync must honor Outlook's `RopSynchronizationConfigure` `PropertyTags` exclusion list for `PidTagContentCount`, `PidTagContentUnreadCount`, `PidTagFolderType`, and `PidTagAccess`. The 2026-05-19 15:43 trace confirmed that forced count/access/type properties plus corrected display names still led Outlook 16.0.19929.20172 to disconnect immediately after hierarchy sync, so continuing to override explicit exclusions is no longer the next supported lab path.

The sync-root `PidTagParentSourceKey` experiment is no longer active. Direct children of the configured hierarchy sync root keep a zero-length `PidTagParentSourceKey`; only deeper descendants carry the real parent folder source key. This is the Microsoft-defined `hierarchySync` shape: [MS-OXCFXICS 2.2.4.3.9](https://learn.microsoft.com/en-us/openspecs/exchange_server_protocols/ms-oxcfxics/dd39f22a-6697-4514-9847-7d3f30a5f12c) requires one `folderChange` for each changed descendant folder of the configured root, and says zero-length `PidTagParentSourceKey` identifies children of that root. ## Canonical Store Adapter Mapping

The MAPI store adapter resolves protocol identifiers and source keys through `mapi_object_identities`, then reads or mutates canonical `LPE` rows. It must not add MAPI-local mailbox content tables. The Microsoft object protocols checked for this adapter on 2026-05-14 were [MS-OXCMSG](https://learn.microsoft.com/en-us/openspecs/exchange_server_protocols/ms-oxcmsg), [MS-OXCFOLD](https://learn.microsoft.com/en-us/openspecs/exchange_server_protocols/ms-oxcfold), and [MS-OXCSTOR](https://learn.microsoft.com/en-us/openspecs/exchange_server_protocols/ms-oxcstor).

| MAPI surface | Canonical `LPE` source of truth | Adapter rule |
| --- | --- | --- |
| Store object and special folders | `accounts`, `mailboxes`, `mapi_mailbox_replicas`, `mapi_object_identities` | Logon and special-folder identifiers are stable MAPI projections over canonical account/mailbox rows; dynamic MAPI global-counter allocation starts after the fixed Outlook special-folder counter range, persisted counters are advanced or repaired if an older lab build allocated inside that reserved range, and MAPI store snapshot bootstrap ensures the canonical system mailbox rows exist before projecting hierarchy sync. |
| Folder open and properties | `mailboxes` | Folder properties derive from canonical mailbox role, display name, counts, and modseq/change facts. |
| Message open and properties | `messages`, `mailbox_messages`, `message_recipients`, `message_bodies` | Normal object access resolves `MID` or source key first, then fetches only the addressed canonical message instead of loading whole mailbox snapshots. |
| Attachment open/write | `attachments`, `mime_parts`, `blobs` through canonical attachment APIs | Attachment streams read/write canonical attachment blobs and update the canonical message attachment state. |
| Recipient table mutation | pending MAPI session state until save, then `message_recipients` / protected `Bcc` persistence through canonical save/import paths | Visible recipients remain visible canonical metadata; `Bcc` stays protected and out of user search/sync projections. |
| Sync checkpoints | `mapi_sync_checkpoints` plus canonical change logs, tombstones, and modseqs | Checkpoints store protocol cursor position only; they are not mailbox content or Outlook-owned state. |

## MAPI Sync State Format

`EMSMDB` ICS download state is serialized as a bounded MS-OXCFXICS FastTransfer stream, not as an `LPE`-private payload. The stream uses the documented synchronization markers `IncrSyncChg`, `IncrSyncDel`, `IncrSyncRead`, `IncrSyncStateBegin`, `IncrSyncStateEnd`, and `IncrSyncEnd`, with message and folder identity projected through `PidTagSourceKey`, `PidTagParentSourceKey`, `PidTagMid`, hierarchy `PidTagFolderId` when `SynchronizationExtraFlags.Eid` is set, content `PidTagChangeNumber` when requested, `PidTagChangeKey`, and `PidTagPredecessorChangeList`. Source keys are stable server GID/XID values over the `LPE` replica GUID and the persisted object global counter from `mapi_object_identities`, not raw eight-byte MAPI folder or message IDs. Change numbers, change keys, predecessor lists, and checkpoint `CNSET` values use monotonic canonical mailbox `modseq` / `mail_change_log` backed counters in the same six-byte `GLOBCNT` domain. Object-id-backed compatibility projections still extract the global counter from the MAPI store ID for stable source and instance keys only. `PidTagPredecessorChangeList` is emitted as serialized PCL data; for current server-originated changes this is a single `SizedXid` containing the same XID-compatible change key. The six-byte `GLOBCNT` portion is serialized in canonical byte-comparison order so values that share a `REPLGUID` increase byte-to-byte, matching the MS-OXCFXICS identity and GLOBSET rules. Outlook 16 cached-mode hierarchy sync also receives required folder change facts such as `PidTagLastModificationTime`, and folder changes serialize the MS-OXCFXICS `folderChange` identity/change properties first, then conditional `PidTagFolderId` / `PidTagParentFolderId`, then unrestricted folder properties such as `PidTagContainerClass`; hierarchy `folderChange` rows do not emit `PidTagChangeNumber` because the required hierarchy change facts are already represented by `PidTagChangeKey`, `PidTagPredecessorChangeList`, and the final hierarchy `MetaTagCnsetSeen` state. Role-backed Outlook special folders use MAPI display-name projections such as `Sent Items` and `Deleted Items` in hierarchy FastTransfer rows, while canonical mailbox names remain unchanged. When Outlook provides `PropertyTags` without `OnlySpecifiedProperties`, those tags are treated as exclusions from the folder-change property list while the stream remains a projection over canonical folder state. Hierarchy `folderChange` rows omit `PidTagContentCount`, `PidTagContentUnreadCount`, `PidTagLocalCommitTimeMax`, empty-folder zero values, and placeholder `PidTagDeletedCountTotal=0` because they are optional folder properties in MS-OXCFXICS `folderChange` rows and current Outlook 16 cached-mode traces still disconnect before content sync when optional hierarchy content-activity facts are present; canonical message counts remain available through folder/table projections and content sync. RCA hierarchy-row logs include both stored mailbox counters and computed snapshot counters, and hierarchy transfer decoder logs include optional-property presence booleans plus the `PropertyTags` filter mode so an omitted optional property is not confused with an emitted zero. Sync configure and upload-state RCA logs also distinguish the baseline transfer size from an available checkpoint-delta transfer and record which buffer is selected after the client uploads synchronization state. Fixed-size FastTransfer values use the MS-OXCFXICS lexical size for their property type, including two-byte `PtypBoolean` values such as `PidTagSubfolders`, so subsequent properties remain atom-aligned for Outlook's FastTransfer parser. Hierarchy sync scopes include changed descendant folders of the configured root, preserve canonical nested mailbox parents through `PidTagParentSourceKey` and `PidTagParentFolderId`, and do not echo the configured root folder as its own `folderChange`; direct children of that synchronization root carry a zero-length `PidTagParentSourceKey`, while deeper descendants carry their real parent folder source key. The hierarchy final `MetaTagIdsetGiven` covers emitted descendant folder source keys, while `MetaTagCnsetSeen` acknowledges both emitted descendant folder changes and the synchronization root change counter so the completed root-scoped hierarchy checkpoint is not narrower than the configured root. `RopSynchronizationConfigure` responses stay exactly on the documented `RopId`, `OutputHandleIndex`, and `ReturnValue` shape so multi-ROP Outlook sync batches do not see padding before the following upload-state or `GetBuffer` response. `RopFastTransferSourceGetBuffer` responses keep `InProgressCount` and `TotalStepCount` as progress-step counters and put only the actual byte length in `TransferBufferSize`, matching MS-OXCFXICS response field semantics.

`RopSynchronizationUploadStateStreamBegin`, `RopSynchronizationUploadStateStreamContinue`, and `RopSynchronizationUploadStateStreamEnd` success responses use the request input handle index, not an output handle index, matching their MS-OXCROPS response shape.

The first local `contentsSync` correctness gate is intentionally narrow: `RopSynchronizationConfigure` with `SynchronizationType` `0x01` over a selected mail folder exports normal mailbox messages from canonical `mailbox_states` as `messageChange` rows only, without hierarchy `folderChange` rows. Each row carries `PidTagSourceKey`, `PidTagParentSourceKey` for the selected folder, `PidTagChangeKey`, `PidTagPredecessorChangeList`, `PidTagAssociated=false`, `PidTagMid` when `Eid` is requested, `PidTagChangeNumber` when `CN` is requested, read/flag properties, subject properties, `PidTagBody` from canonical message body text when requested or not excluded, visible recipient facts, and attachment facts without protected `Bcc`. Common Views content sync also exports read-only FAI `messageChange` rows for canonical search-folder definition messages with `PidTagAssociated=true`, stable definition-message source keys, `IPM.Microsoft.WunderBar.SFInfo`, and the bounded `PidTagSearchFolderDefinition` projection used by the associated contents table. Conversation Action Settings content sync exports canonical `IPM.ConversationAction` FAI rows with `PidTagAssociated=true`, `PidTagConversationIndex`, conversation-action LID properties, and `PidNameKeywords`; incremental sync also exports destroyed conversation-action FAI rows as `IncrSyncDel`. Incremental content sync can export hard-delete or moved-out tombstones as `IncrSyncDel` with `MetaTagIdsetDeleted`, and when `ReadState` is requested it emits one `IncrSyncRead` section with `MetaTagIdsetRead` and/or `MetaTagIdsetUnread` over message IDs. The final content state includes `MetaTagIdsetGiven`, `MetaTagCnsetSeen`, `MetaTagCnsetSeenFAI`, and `MetaTagCnsetRead` over message IDs and canonical message change numbers. This gate is enough for local harness validation that Outlook should be able to parse and download INBOX and bounded FAI rows after it begins issuing content sync; it is not yet evidence that real Outlook 2016 / 2019 cached mode completes the profile.

Known gaps in the first content-sync gate remain: restrictions, conflict detection, partial item downloads, full body stream/rich-text projections beyond canonical plain text, FAI content beyond Common Views search-folder definitions, full recipient and attachment FastTransfer object graphs, optimized read-state-only deltas, server-side parsing of client-provided `IDSET` / `CNSET` details beyond checkpoint selection, cross-process session replay, and real Outlook close/reopen cache stability are not complete.

Final and checkpoint state is represented with the ICS state meta-properties `MetaTagIdsetGiven`, `MetaTagCnsetSeen`, `MetaTagCnsetSeenFAI`, and `MetaTagCnsetRead` as REPLGUID-scoped serialized IDSET/GLOBSET payloads over the canonical `LPE` replica GUID. Their GLOBSET range commands carry canonical six-byte `GLOBCNT` values rather than native-endian integers. Hierarchy sync final `MetaTagIdsetGiven` remains scoped to the folder IDs actually emitted in the hierarchy transfer, while final `MetaTagCnsetSeen` covers those emitted folder change numbers plus the configured synchronization root change counter; message IDs and message change counters belong to content sync state only. Content sync and folder object lookups select messages from canonical per-folder `mailbox_states`, with the primary mailbox fields kept only as a compatibility fallback for legacy projections. Message change numbers include those per-folder membership and state facts so moves, copies, and folder-local read-state changes advance the MAPI `CNSET`. Folder `PidTagSourceKey` values in hierarchy downloads use the same projected MAPI folder object IDs that conditionally appear in `PidTagFolderId` and always appear in `MetaTagIdsetGiven`, including fixed IDs for Outlook special folders such as `Inbox`, `Drafts`, and `Sent`; mailbox UUID mappings remain canonical storage identities, not separate hierarchy ICS identities. Private-store hierarchy sync also projects the non-canonical special folders advertised by the `RopLogon` private mailbox `FolderIds` list, such as `Outbox`, `Common Views`, `Schedule`, `Search`, `Views`, and `Shortcuts`, when they are descendants of the requested sync root. The IPM subtree hierarchy additionally projects Outlook default IPM special folders that are identified outside the `RopLogon` folder-id list, including mandatory cached-mode bootstrap folders `Calendar`, `Contacts`, `Journal`, `Notes`, and `Tasks`; these folders must have stable folder IDs, source keys, hierarchy rows, `OpenFolder` behavior, `GetProps` rows, and default entry ID mappings through `PidTagIpmAppointmentEntryId`, `PidTagIpmContactEntryId`, `PidTagIpmJournalEntryId`, `PidTagIpmNoteEntryId`, and `PidTagIpmTaskEntryId`. Those default-folder identification properties are folder `EntryID` values, not bare 24-byte LongTermIDs; the write path accepts both shapes for compatibility, but generated `GetProps` values use the full account-scoped folder EntryID. Optional/search compatibility folders keep stable IDs for direct-open and store-property compatibility, but they are not emitted as visible IPM subtree `folderChange` rows and are not acknowledged in hierarchy final state until real Outlook traces validate each row shape. The 2026-05-21 16:14 RCA trace showed Outlook disconnecting immediately after a hierarchy transfer whose final state acknowledged 26 stable IPM counters while only 19 folder-change rows were emitted; however, the 2026-05-21 16:24 RCA trace returned to the row-scoped 19-folder final state (`MetaTagIdsetGiven` 95 bytes, `MetaTagCnsetSeen` 121 bytes) and still disconnected before content sync, so the 26-counter state was not the primary blocker. The 2026-05-21 20:31 trace advanced through hierarchy sync, `RopGetReceiveFolder`, Inbox default-folder `GetProps`, and root `SetProperties`, then disconnected before content sync; that trace exposed the remaining mismatch that the six `PidTagIpm*EntryId` / `PidTagRemOnlineEntryId` values were emitted as 24-byte LongTermIDs instead of folder EntryIDs. The matching 16:24 Outlook ETL showed provider `{aa8fa310-0939-4ce3-b9bb-ae05b2695110}` event `4001` with `0x80040102` immediately after the preceding single-ROP `RopGetPropertiesSpecific` EMSMDB `Execute` (`client_request_id={4B4E9BBB-4D38-46B6-A986-C78F1A41651B}`), followed by event `7001` with `0x80004005` after the hierarchy `Execute` response (`client_request_id={417A26AB-CBF9-4341-BD1D-94FD2333A95D}`); this makes the pre-hierarchy store/logon property response the next investigation target. RCA logging for that Outlook bootstrap `GetProps` now decodes the mailbox owner PermanentEntryID fields, user GUID bytes, store-state value, private flag, and server icon ICO headers so the next trace can distinguish malformed property payloads from a later hierarchy/content-sync issue. Reminders is exposed through `PidTagRemOnlineEntryId` and direct `OpenFolder` as a computed search-folder projection over canonical reminder-bearing appointments, tasks, and mail messages; it is not a normal hierarchy folder row and has no protocol-local backing store. These folders are emitted in fixed Outlook special-folder order before falling back to display-name ordering. These rows are virtual MAPI hierarchy projections only; they do not create mailbox, collaboration, or user-visible state outside the canonical `LPE` store. Deleted or moved-out messages are exported as `IncrSyncDel` with `MetaTagIdsetDeleted` as a REPLID-scoped serialized IDSET/GLOBSET over the MAPI message IDs derived from canonical identity mappings.

`mapi_sync_checkpoints` stores only the durable cursor needed to resume a synchronization scope: `checkpoint_kind`, optional `mailbox_id`, the MAPI replica GUID, last canonical change sequence, last mail modseq, and a small JSON cursor descriptor. Hierarchy checkpoints are account-wide but are usable only for the same hierarchy sync root folder and hierarchy sync cursor version stored in the cursor metadata. Content and read-state checkpoints are mailbox-scoped. On `RopSynchronizationConfigure`, `LPE` reads the checkpoint and replays `mail_change_log` plus `tombstones` after the stored cursor; after `RopFastTransferSourceGetBuffer` drains the ICS download stream, it advances the checkpoint. A usable server checkpoint selects this canonical delta only after the client uploads non-empty synchronization state; zero-length uploaded state keeps the full baseline transfer so a fresh Outlook local replica is not asked to apply a delta over missing folders. Uploaded client ICS state is treated only as synchronization input and must not be appended to the server-generated final/checkpoint state returned by `RopSynchronizationGetTransferState` or the transfer stream. Transfer-state handles created from an ICS download source must retain that source's canonical checkpoint sequence and modseq so draining the derived stream cannot regress `mapi_sync_checkpoints`; transfer-state handles created from collector/import contexts must not advance download checkpoints at all. Upload/import ROPs mutate canonical mailbox state and rely on the same change-log/tombstone path for subsequent download reconciliation.

## Table Projection Support

The MAPI table engine uses MS-OXCTABL cursor semantics and MS-OXCDATA `StandardPropertyRow`, `Restriction`, and `SortOrder` structures while keeping table rows as projections over canonical `LPE` data.

## Acceptance Criteria

- Every implemented MAPI behavior in this document maps to one or more Microsoft specification sections.
- New MAPI behavior must add tests for the request type, ROP group, identity/property/sync/session behavior, or deferred parseability it changes.
- Unsupported typed enum values must be logged with their raw value and verified through the existing parseable unsupported/error paths rather than through panics or generic runtime failures.
