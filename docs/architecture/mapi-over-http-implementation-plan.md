# MAPI over HTTP Implementation Plan

## Current State/Functionality Overview

This document is the Microsoft specification-to-`LPE` implementation reference for the guarded MAPI over HTTP compatibility track.

The reference is documentation-only. It does not change runtime behavior. `LPE` remains the canonical store for mailboxes, contacts, calendars, tasks, rights, submission, and user-visible state, while MAPI over HTTP stays an authenticated Outlook desktop compatibility surface over that canonical state.

The initial ticket sequence has been completed and removed from this document. The remaining matrices describe implemented coverage, release gates, and deferred surfaces.

The Microsoft protocol specifications were checked on 2026-05-17 for `MS-OXCFXICS` `RopSynchronizationConfigure`, `hierarchySync`, `contentsSync`, `messageChange`, read-state changes, deletion state, final ICS state, `IDSET` / `CNSET` syntax, and FastTransfer property lexical encoding. The cited specifications define wire contracts and data structures, but they do not define Outlook's exact tolerance for partial Exchange compatibility. Real Outlook 2016 and Outlook 2019 profile creation, cached-mode synchronization, reconnect, and RCA evidence remain required release gates.

MAPI over HTTP is the first Outlook desktop Exchange-account path for this work. Outlook Anywhere / RPC over HTTP is a legacy compatibility shim for later `EXPR` publication and must not drive the initial MAPI over HTTP implementation order.

Single-node sticky MAPI sessions are acceptable for the first Outlook 2016 / 2019 lab gate. Cross-process session replay, durable session migration, and load-balanced session failover remain production hardening.

Readiness terminology is intentionally split:

- Local harness pass means `crates/lpe-exchange` tests and project-owned live scripts, including `tools/rca_outlook_connectivity_check.py --outlook-rca-readiness`, pass against a configured deployment. This proves LPE's scripted compatibility path, not public Outlook readiness by itself.
- RCA pass means Microsoft Remote Connectivity Analyzer Outlook Connectivity passes from the Internet against the same public host and account that will be advertised.
- Real Outlook profile pass means Outlook 2016 and Outlook 2019 each create an Exchange account profile, complete cached-mode synchronization, close and reopen without a full-cache wipe, resolve NSPI entries, send through canonical submission, and show the authoritative `Sent` copy.

Current MAPI status: local harness and server-side hierarchy-sync traces have passed through the stages recorded below, but the full publication gate remains pending until Microsoft RCA evidence and real Outlook 2016 / 2019 cached-mode profile evidence are both recorded.

Successful EMSMDB `Execute` responses refresh both the `MapiContext` session cookie and the matching `MapiSequence` cookie so request replay and later session-bound requests can continue with a complete single-node in-process session context.

Reserved or otherwise unsupported ROPs are terminal within the current ROP request buffer: the server returns one parseable unsupported ROP error and does not execute later ROP bytes from that batch.

`RopLogon` creates a store handle only for private mailbox logons. Public-folder logons, identified by the request `Private` flag not being set, remain deferred with a parseable unsupported ROP response and no protocol-local public-folder store state.

## Outlook Hierarchy Count Interoperability Experiment

`LPE_MAPI_EXPERIMENT_FORCE_HIERARCHY_COUNT_PROPERTIES` is an opt-in MAPI/HTTP experiment and is disabled by default. When set to `true`, `1`, `yes`, or `on`, authenticated EMSMDB hierarchy sync emits `PidTagContentCount` and `PidTagContentUnreadCount` from snapshot-computed folder counts even when Outlook supplied those tags in the `RopSynchronizationConfigure` `PropertyTags` exclusion list. The same experiment also keeps `PidTagFolderType` and `PidTagAccess` in the hierarchy row because the 2026-05-18 trace reached the default-folder bootstrap but still disconnected before content sync with both of those bootstrap properties absent. The default path still treats `PropertyTags` without `OnlySpecifiedProperties` as exclusions.

The sync-root `PidTagParentSourceKey` experiment is no longer active. Direct children of the configured hierarchy sync root keep a zero-length `PidTagParentSourceKey`; only deeper descendants carry the real parent folder source key.

Current Outlook lab matrix:

| ParentSourceKey mode | Hierarchy count mode | Outlook evidence |
| --- | --- | --- |
| Empty for direct sync-root children; `PidTagFolderId` omitted unless `Eid` is requested | `LPE_MAPI_EXPERIMENT_FORCE_HIERARCHY_COUNT_PROPERTIES=true` | 2026-05-18 17:36 trace: clean hierarchy FastTransfer, then client `Disconnect` before any post-hierarchy Execute. Outlook ETL records provider `{aa8fa310-0939-4ce3-b9bb-ae05b2695110}` error events carrying `0x80040102` after the preceding EMSMDB `Execute` and `0x80004005` after the hierarchy `Execute`, without decoded ROP names. |
| Empty for direct sync-root children; pre-conditional `PidTagFolderId` stream | `LPE_MAPI_EXPERIMENT_FORCE_HIERARCHY_COUNT_PROPERTIES=true` | Reached post-hierarchy default-folder/bootstrap probes, then disconnected. |
| Non-empty for direct sync-root children | `LPE_MAPI_EXPERIMENT_FORCE_HIERARCHY_COUNT_PROPERTIES=true` | Immediate disconnect. |
| Empty for direct sync-root children | Forced counts disabled | Immediate disconnect. |

The 2026-05-18 forced-count trace shows the post-hierarchy sequence `RopGetReceiveFolder`, then `RopOpenFolder` on `Inbox` with `RopGetPropertiesSpecific` for `PidTagIpmAppointmentEntryId`, `PidTagIpmContactEntryId`, `PidTagIpmJournalEntryId`, `PidTagIpmNoteEntryId`, `PidTagIpmTaskEntryId`, and `PidTagRemOnlineEntryId`, then `RopOpenFolder` on the hierarchy root plus `RopSetProperties` for the same six properties. Each property value is a 24-byte folder long-term ID, and the `RopSetProperties` response returns success with `property_problem_count=0`. Outlook then sends client `Disconnect` without any content-sync configure. The matching Outlook ETL decoded with `tracerpt` does not expose ROP names, but it does show Outlook provider `{aa8fa310-0939-4ce3-b9bb-ae05b2695110}` level-2 events on the same client activity GUID at this stop point with HRESULT-like values `0x80040102` and then `0x80004005`.

The six root default-folder properties accepted through that `RopSetProperties` path are retained in the EMSMDB session's root-folder property state and persisted in the MAPI folder-property store. Later folder opens hydrate persisted client-set folder properties before serving `RopGetPropertiesSpecific`; the root session cache still overlays the just-set values inside the active session. This lets RCA logs distinguish Outlook-provided stored values from computed compatibility defaults across reconnects.

Rationale: current Outlook 16 traces show a valid hierarchy ICS stream followed by disconnect before content sync. The affected mailbox has snapshot-computed `Inbox` content and unread counts, but Outlook excluded the count properties. This experiment tests whether Outlook's cached-mode bootstrap expects those folder aggregate properties despite sending them as exclusions. This is not a protocol MUST and must not become the default without real Outlook evidence.

RCA evidence requirements:

- `rca debug mapi sync configure` records `force_hierarchy_count_properties_experiment`.
- `rca debug mapi hierarchy row` records `hierarchy_count_experiment_enabled`, `content_count_forced_by_experiment`, `content_unread_count_forced_by_experiment`, `folder_type_forced_by_experiment`, and `access_forced_by_experiment`.
- `rca debug mapi hierarchy transfer row semantics` must show count, folder-type, and access property presence when the experiment forces them.

Rollback condition: remove or keep disabled by default if Outlook still disconnects after hierarchy sync, if Microsoft RCA regresses, if real Outlook profile creation does not proceed to content synchronization, or if any client/parser rejects the hierarchy FastTransfer stream with the forced properties present.

Outlook A/B lab:

1. A disabled: leave `LPE_MAPI_EXPERIMENT_FORCE_HIERARCHY_COUNT_PROPERTIES` unset or set it to `false`, restart the core service, create a fresh Outlook 2016 or Outlook 2019 cached-mode profile, and capture server RCA logs plus Outlook status.
2. B enabled: set `LPE_MAPI_EXPERIMENT_FORCE_HIERARCHY_COUNT_PROPERTIES=true`, restart the core service, create another fresh cached-mode profile against the same mailbox state, and capture the same logs.
3. Expected success signal: after hierarchy sync, Outlook issues a content synchronization request (`RopSynchronizationConfigure` with `SynchronizationType` `0x01`) instead of disconnecting after the hierarchy FastTransfer stream.
4. If B succeeds, repeat close/reopen and initial sync with the flag still enabled, then record whether canonical `Sent` visibility and reconnect behavior remain unchanged.

`PidTagMaxMessageObjectId` was searched as requested, but no Microsoft Open Specifications property with that canonical name was found in the current [MS-OXPROPS] master property list. `LPE` must not invent or persist such a property by name. If Outlook 2016 / 2019 lab traces show a concrete property tag for "max message object id", that tag becomes a lab-derived engineering requirement and must be added here with the trace evidence before implementation.

Outlook 2016 / 2019 lab expectations derived from this baseline:

- Cached-mode profile creation must be able to fetch stable `PidTagSourceKey`, `PidTagChangeKey`, `PidTagPredecessorChangeList`, `PidTagLocalCommitTime`, and folder hierarchy revision properties from folder/message property requests and ICS manifests.
- First-run ICS uploads may provide zero-length `MetaTagIdsetGiven` / CNSET state; repeat runs must use the client-supplied state to avoid full redownloads.
- Reconnect and repeated `Execute` request-id replay must not allocate new source keys, MIDs, change keys, or hierarchy revisions for unchanged canonical objects.
- Lab pass criteria: Outlook 2016 and Outlook 2019 cached-mode profiles complete initial sync, close/reopen without a full-cache wipe, receive one server-side message and one hierarchy change without loop-back reimport, and keep `Sent` visibility canonical through the existing cross-protocol gate.

## Canonical Store Adapter Mapping

The MAPI store adapter resolves protocol identifiers and source keys through `mapi_object_identities`, then reads or mutates canonical `LPE` rows. It must not add MAPI-local mailbox content tables. The Microsoft object protocols checked for this adapter on 2026-05-14 were [MS-OXCMSG](https://learn.microsoft.com/en-us/openspecs/exchange_server_protocols/ms-oxcmsg), [MS-OXCFOLD](https://learn.microsoft.com/en-us/openspecs/exchange_server_protocols/ms-oxcfold), and [MS-OXCSTOR](https://learn.microsoft.com/en-us/openspecs/exchange_server_protocols/ms-oxcstor).

| MAPI surface | Canonical `LPE` source of truth | Adapter rule |
| --- | --- | --- |
| Store object and special folders | `accounts`, `mailboxes`, `mapi_mailbox_replicas`, `mapi_object_identities` | Logon and special-folder identifiers are stable MAPI projections over canonical account/mailbox rows; dynamic MAPI global-counter allocation starts after the fixed Outlook special-folder counter range, persisted counters are advanced or repaired if an older lab build allocated inside that reserved range, and MAPI store snapshot bootstrap ensures the canonical system mailbox rows exist before projecting hierarchy sync. |
| Folder open and properties | `mailboxes` | Folder properties derive from canonical mailbox role, display name, counts, and modseq/change facts. |
| Message open and properties | `messages`, `mailbox_messages`, `message_recipients`, `message_bodies` | Normal object access resolves `MID` or source key first, then fetches only the addressed canonical message instead of loading whole mailbox snapshots. |
| Attachment open/write | `attachments`, `mime_parts`, `blobs` through canonical attachment APIs | Attachment streams read/write canonical attachment blobs and update the canonical message attachment state. |
| Recipient table mutation | pending MAPI session state until save, then `message_recipients` / protected `Bcc` persistence through canonical save/import paths | Visible recipients remain visible canonical metadata; `Bcc` stays protected and out of user search/sync projections. |
| Sync checkpoints | `mapi_sync_checkpoints` plus canonical change logs, tombstones, and modseqs | Checkpoints store protocol cursor position only; they are not mailbox content or Outlook-owned state. |

## MAPI Sync State Format

`EMSMDB` ICS download state is serialized as a bounded MS-OXCFXICS FastTransfer stream, not as an `LPE`-private payload. The stream uses the documented synchronization markers `IncrSyncChg`, `IncrSyncDel`, `IncrSyncRead`, `IncrSyncStateBegin`, `IncrSyncStateEnd`, and `IncrSyncEnd`, with message and folder identity projected through `PidTagSourceKey`, `PidTagParentSourceKey`, `PidTagMid`, conditionally emitted `PidTagFolderId` values when the client requests the `Eid` synchronization extra flag, `PidTagChangeNumber`, `PidTagChangeKey`, and `PidTagPredecessorChangeList`. Source keys are stable server GID/XID values over the `LPE` replica GUID and the persisted object global counter from `mapi_object_identities`, not raw eight-byte MAPI folder or message IDs. Change numbers, change keys, predecessor lists, and checkpoint `CNSET` values use monotonic canonical mailbox `modseq` / `mail_change_log` backed counters in the same six-byte `GLOBCNT` domain. Object-id-backed compatibility projections still extract the global counter from the MAPI store ID for stable source and instance keys only. `PidTagPredecessorChangeList` is emitted as serialized PCL data; for current server-originated changes this is a single `SizedXid` containing the same XID-compatible change key. The six-byte `GLOBCNT` portion is serialized in canonical byte-comparison order so values that share a `REPLGUID` increase byte-to-byte, matching the MS-OXCFXICS identity and GLOBSET rules. Outlook 16 cached-mode hierarchy sync also receives required folder change facts such as `PidTagLastModificationTime`, and folder changes serialize the required source-key, change, display-name, and `PidTagContainerClass` folder-class properties in MS-OXCFXICS folder-change order before conditional folder IDs and other properties; when Outlook provides `PropertyTags` without `OnlySpecifiedProperties`, those tags are treated as exclusions from the folder-change property list while the stream remains a projection over canonical folder state. When hierarchy folder count properties are emitted, `PidTagContentCount` and `PidTagContentUnreadCount` are derived from the loaded canonical message snapshot so virtual MAPI special folders do not advertise stale zero counts while messages are present; `PidTagLocalCommitTimeMax` uses the same per-folder message membership instead of only an email's primary mailbox. RCA hierarchy-row logs include both stored mailbox counters and computed snapshot counters, and hierarchy transfer decoder logs include optional-property presence booleans plus the `PropertyTags` filter mode so an omitted optional property is not confused with an emitted zero. Sync configure and upload-state RCA logs also distinguish the baseline transfer size from an available checkpoint-delta transfer and record which buffer is selected after the client uploads synchronization state. Fixed-size FastTransfer values use the MS-OXCFXICS lexical size for their property type, including two-byte `PtypBoolean` values such as `PidTagSubfolders`, so subsequent properties remain atom-aligned for Outlook's FastTransfer parser. Hierarchy sync scopes include changed descendant folders of the configured root, preserve canonical nested mailbox parents through `PidTagParentSourceKey` and `PidTagParentFolderId`, and do not echo the configured root folder as its own `folderChange`; direct children of that synchronization root carry a zero-length `PidTagParentSourceKey`, while deeper descendants carry their real parent folder source key. `RopSynchronizationConfigure` responses stay exactly on the documented `RopId`, `OutputHandleIndex`, and `ReturnValue` shape so multi-ROP Outlook sync batches do not see padding before the following upload-state or `GetBuffer` response. `RopFastTransferSourceGetBuffer` responses keep `InProgressCount` and `TotalStepCount` as progress-step counters and put only the actual byte length in `TransferBufferSize`, matching MS-OXCFXICS response field semantics.

The first local `contentsSync` correctness gate is intentionally narrow: `RopSynchronizationConfigure` with `SynchronizationType` `0x01` over a selected mail folder exports normal mailbox messages from canonical `mailbox_states` as `messageChange` rows only, without hierarchy `folderChange` rows. Each row carries `PidTagSourceKey`, `PidTagParentSourceKey` for the selected folder, `PidTagChangeKey`, `PidTagPredecessorChangeList`, `PidTagAssociated=false`, `PidTagMid` when `Eid` is requested, `PidTagChangeNumber` when `CN` is requested, read/flag properties, subject properties, visible recipient facts, and attachment facts without protected `Bcc`. Incremental content sync can export hard-delete or moved-out tombstones as `IncrSyncDel` with `MetaTagIdsetDeleted`, and when `ReadState` is requested it emits one `IncrSyncRead` section with `MetaTagIdsetRead` and/or `MetaTagIdsetUnread` over message IDs. The final content state includes `MetaTagIdsetGiven`, `MetaTagCnsetSeen`, `MetaTagCnsetSeenFAI`, and `MetaTagCnsetRead` over message IDs and canonical message change numbers. This gate is enough for local harness validation that Outlook should be able to parse and download INBOX message rows after it begins issuing content sync; it is not yet evidence that real Outlook 2016 / 2019 cached mode completes the profile.

Known gaps in the first content-sync gate remain: FAI content, restrictions, conflict detection, partial item downloads, per-property client include/exclude behavior for message bodies, full recipient and attachment FastTransfer object graphs, optimized read-state-only deltas, server-side parsing of client-provided `IDSET` / `CNSET` details beyond checkpoint selection, cross-process session replay, and real Outlook close/reopen cache stability are not complete.

Final and checkpoint state is represented with the ICS state meta-properties `MetaTagIdsetGiven`, `MetaTagCnsetSeen`, `MetaTagCnsetSeenFAI`, and `MetaTagCnsetRead` as REPLGUID-scoped serialized IDSET/GLOBSET payloads over the canonical `LPE` replica GUID. Their GLOBSET range commands carry canonical six-byte `GLOBCNT` values rather than native-endian integers. Hierarchy sync final state remains scoped to folder IDs and folder change numbers, even when folder-change rows include aggregate content properties such as `PidTagLocalCommitTimeMax` and `PidTagDeletedCountTotal`; message IDs and message change counters belong to content sync state only. Content sync and folder object lookups select messages from canonical per-folder `mailbox_states`, with the primary mailbox fields kept only as a compatibility fallback for legacy projections. Message change numbers include those per-folder membership and state facts so moves, copies, and folder-local read-state changes advance the MAPI `CNSET`. Folder `PidTagSourceKey` values in hierarchy downloads use the same projected MAPI folder object IDs that appear in `PidTagFolderId` and `MetaTagIdsetGiven`, including fixed IDs for Outlook special folders such as `Inbox`, `Drafts`, and `Sent`; mailbox UUID mappings remain canonical storage identities, not separate hierarchy ICS identities. Private-store hierarchy sync also projects the non-canonical special folders advertised by the `RopLogon` private mailbox `FolderIds` list, such as `Outbox`, `Common Views`, `Schedule`, `Search`, `Views`, and `Shortcuts`, when they are descendants of the requested sync root. The IPM subtree hierarchy additionally projects Outlook default IPM special folders that are identified outside the `RopLogon` folder-id list, including mandatory cached-mode bootstrap folders `Calendar`, `Contacts`, `Journal`, `Notes`, `Tasks`, and `Reminders`; these folders must have stable folder IDs, source keys, hierarchy rows, `OpenFolder` behavior, `GetProps` rows, and default entry ID mappings through `PidTagIpmAppointmentEntryId`, `PidTagIpmContactEntryId`, `PidTagIpmJournalEntryId`, `PidTagIpmNoteEntryId`, `PidTagIpmTaskEntryId`, and the reminder online entry ID. These folders are emitted in fixed Outlook special-folder order before falling back to display-name ordering. These rows are virtual MAPI hierarchy projections only; they do not create mailbox, collaboration, or user-visible state outside the canonical `LPE` store. Deleted or moved-out messages are exported as `IncrSyncDel` with `MetaTagIdsetDeleted` as a REPLID-scoped serialized IDSET/GLOBSET over the MAPI message IDs derived from canonical identity mappings.

`mapi_sync_checkpoints` stores only the durable cursor needed to resume a synchronization scope: `checkpoint_kind`, optional `mailbox_id`, the MAPI replica GUID, last canonical change sequence, last mail modseq, and a small JSON cursor descriptor. Hierarchy checkpoints are account-wide but are usable only for the same hierarchy sync root folder and hierarchy sync cursor version stored in the cursor metadata. Content and read-state checkpoints are mailbox-scoped. On `RopSynchronizationConfigure`, `LPE` reads the checkpoint and replays `mail_change_log` plus `tombstones` after the stored cursor; after `RopFastTransferSourceGetBuffer` drains the ICS download stream, it advances the checkpoint. A usable server checkpoint selects this canonical delta only after the client uploads non-empty synchronization state; zero-length uploaded state keeps the full baseline transfer so a fresh Outlook local replica is not asked to apply a delta over missing folders. Uploaded client ICS state is treated only as synchronization input and must not be appended to the server-generated final/checkpoint state returned by `RopSynchronizationGetTransferState` or the transfer stream. Transfer-state handles created from an ICS download source must retain that source's canonical checkpoint sequence and modseq so draining the derived stream cannot regress `mapi_sync_checkpoints`; transfer-state handles created from collector/import contexts must not advance download checkpoints at all. Upload/import ROPs mutate canonical mailbox state and rely on the same change-log/tombstone path for subsequent download reconciliation.

## Table Projection Support

The MAPI table engine uses MS-OXCTABL cursor semantics and MS-OXCDATA `StandardPropertyRow`, `Restriction`, and `SortOrder` structures while keeping table rows as projections over canonical `LPE` data.

| Table behavior | Supported scope | Storage behavior |
| --- | --- | --- |
| Hierarchy tables | Root and IPM subtree child folders, including canonical mail folders and collaboration folders. | Uses mailbox and collaboration collection projections; no mailbox-message scan is required. |
| Hierarchy table ordering | Canonical mailbox folders and collaboration folders are sorted as one projected row set when `RopSortTable` is active. | Prevents folder-family grouping from overriding caller-selected table sort order; no separate MAPI folder store is introduced. |
| Contents tables | Mail, contact, calendar, and task contents projections with caller-selected columns. | Mail contents `QueryRows` without a restriction and with supported sort keys uses a paged PostgreSQL ID query before fetching only returned rows. Contacts, calendar, and task contents remain snapshot-backed over canonical collaboration state. |
| Attachment tables | Canonical message attachment rows with caller-selected columns. | Uses the addressed message attachment set only. |
| Permission tables | Folder permission rows for `Default`, `Anonymous`, mailbox owner, and canonical mailbox delegation grantees. Supported columns are `PidTagMemberId`, `PidTagMemberName`, and `PidTagMemberRights`. | Uses existing `mailbox_delegation_grants`; no Exchange-specific ACL store is added. |
| Supported SQL-backed mail sorts | `PidTagMessageDeliveryTime`, `PidTagLastModificationTime`, `PidTagSubject`, `PidTagNormalizedSubject`, `PidTagSenderName`, `PidTagSenderEmailAddress`, `PidTagDisplayTo`, `PidTagMessageSize`, `PidTagHasAttachments`, and `PidTagMessageFlags`, ascending or descending. | Sorts are translated to PostgreSQL `ORDER BY` with a stable message-id tie-breaker. |
| Restrictions | MS-OXCDATA `AND`, `OR`, `NOT`, content, property, bitmask, size, and exist restrictions parse and evaluate for implemented projected properties. | Restricted mail contents currently use the bounded in-memory fallback because only a subset of arbitrary restriction trees maps cleanly to SQL. Unsupported restriction forms return protocol errors instead of being coerced. |
| Bookmarks | Contents and hierarchy bookmarks remember stable row identity plus cursor offset. | Sort, restrict, and reset invalidate bookmarks; if a bookmarked row is no longer visible, seek reports that condition and moves from the stored offset. |

## Acceptance Criteria

- Every implemented MAPI behavior in this document maps to one or more Microsoft specification sections.
- New MAPI behavior must add tests for the request type, ROP group, identity/property/sync/session behavior, or deferred parseability it changes.
- New MAPI behavior must preserve canonical `LPE` submission, `Sent`, mailbox, contact, calendar, task, rights, and protected `Bcc` boundaries.
- MAPI mutation changes for send, draft, move, copy, delete, flags, attachments, or protected recipients must pass the cross-protocol storage gate proving IMAP and JMAP observe the same canonical state, including no MAPI-only `Sent`, `Outbox`, search, or sync state.
- Public `mapiHttp` and MAPI-backed `EXCH` publication remains gated by the existing architecture and client-autoconfiguration rules, including `LPE_AUTOCONFIG_OUTLOOK_INTEROP_GATE_PASSED`.
- Legacy `EXPR` publication additionally requires the later RPC/HTTP compatibility shim gate and must not block the first MAPI over HTTP implementation path.
