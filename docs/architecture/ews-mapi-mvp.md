# EWS and MAPI

## Current State/Functionality Overview

`lpe-exchange` exposes bounded EWS compatibility and guarded MAPI over HTTP endpoints over canonical `LPE` mailbox, contacts, calendar, task, address-book, and submission state. Full Outlook functionality is the target, but current Exchange compatibility remains incomplete and release-gated; the adapter must not introduce Exchange-specific mailbox storage or protocol-local canonical state.

The detailed Microsoft specification-to-`LPE` implementation matrix for MAPI over HTTP is maintained in `docs/architecture/mapi-over-http-implementation-plan.md`.

## Implementation/Usage

- EWS endpoints:
  - `OPTIONS /EWS/Exchange.asmx`
  - `POST /EWS/Exchange.asmx`
  - `OPTIONS /ews/exchange.asmx`
  - `POST /ews/exchange.asmx`
- MAPI over HTTP endpoints:
  - `OPTIONS /mapi/emsmdb`
  - `POST /mapi/emsmdb`
  - `OPTIONS /mapi/nspi`
  - `POST /mapi/nspi`
- MAPI implementation modules under `crates/lpe-exchange/src/mapi/`:
  - `transport.rs` owns MAPI/HTTP request validation, response envelopes, cookies, diagnostics, and endpoint routing.
  - `session.rs` owns authenticated session state, handle tables, request replay caches, and, for the later legacy shim, RPC/HTTP EMSMDB context execution.
  - `dispatch.rs` owns Execute request decoding and ROP dispatch against canonical `LPE` state.
  - `rop.rs` owns ROP buffer parsing, ROP response encoding, and low-level cursor helpers.
  - `tables.rs` owns hierarchy, contents, attachment, contact, and calendar table projections.
  - `properties.rs` owns property tags, named properties, value conversion, streams, and canonical property application.
  - `sync.rs` owns replica identifiers, FastTransfer/ICS buffers, manifests, and sync object mapping.
  - `nspi.rs` owns address-book request handling and tenant-visible NSPI projections.
- Outlook Anywhere / RPC over HTTP legacy-shim endpoint:
  - `/rpc/rpcproxy.dll`
  - `/RPC/RpcProxy.dll` as a case-compatible direct-core alias for Outlook/RCA probes; it uses the same authenticated handler and does not publish a separate protocol surface
- Authentication:
  - mailbox account authentication
  - authenticated endpoints only
- EWS adapter rules:
  - reads and writes canonical mailbox, contacts, calendar, and task state
  - contact updates preserve canonical rich fields, vCard/source metadata, and unsupported structured arrays when an EWS request omits them; explicit delete/update instructions clear only the addressed fields
  - treats Outlook-visible Suggested Contacts as the canonical `contact_books.role = 'suggested_contacts'` folder containing durable `contacts`
  - does not expose private `recipient_suggestions` rows as contacts, directory entries, shared contact grants, search results, or AI-facing data
  - uses canonical submission for sending
  - records authoritative `Sent`
  - never implements client `SMTP`
  - never stores Exchange-only mailbox state
- MAPI rules:
  - `EMSMDB` maps mailbox synchronization to canonical mailbox state
  - `NSPI` maps address-book behavior to canonical account/contact visibility
  - contact property updates merge into canonical contacts and do not erase rich contact fields, vCard/source metadata, or structured arrays that the MAPI property set did not address
  - recipient autocomplete suggestions are owner-scoped compose assistance and must not become NSPI address-book rows; `Bcc` recipients are never learned or projected as suggestions
  - session context must remain authenticated and bounded to the mailbox principal
  - send and draft flows must use canonical submission and draft persistence
  - profile creation must complete authenticated `Connect`, private mailbox `Logon`, hierarchy/content table probes, and NSPI address-book binding without publishing MAPI autodiscover until the gate passes
  - mailbox synchronization must use canonical message, folder, flag, attachment, and sync-state mappings; import/read/delete/move operations must mutate canonical state only
- Autodiscover:
  - `EWS` publication requires `LPE_AUTOCONFIG_EWS_ENABLED`
  - `mapiHttp` publication requires `LPE_AUTOCONFIG_MAPI_ENABLED` and a supported positive Outlook `X-MapiHttpCapability` probe; 0.5.0 new installations enable the deployment flag after the successful profile/sync/reconnect milestone, while live RCA and real Outlook evidence remain release-quality checks rather than a second runtime switch
  - legacy `EXCH` publication requires `LPE_AUTOCONFIG_EXCH_AUTODISCOVER_ENABLED`
  - legacy `EXPR` publication requires `LPE_AUTOCONFIG_EXPR_AUTODISCOVER_ENABLED`, `LPE_AUTOCONFIG_RPC_PROXY_ENABLED`, and `LPE_AUTOCONFIG_OUTLOOK_INTEROP_GATE_PASSED`; this is a later legacy compatibility gate and not the first MAPI over HTTP publication path

| Associated contents and `RopFindRow` | Folder-associated contents tables expose persisted FAI messages plus the canonical folder-local default named view. `RopFindRow` applies its request restriction only while searching the table's existing rowset, moves only the cursor when a real row matches, and returns `ecNotFound` when no row matches. | Durable folder version identity | Each reserved private-store folder has one durable `mapi_object_identities` version tuple `(CN, ChangeKey, PCL, LMT)`. The FID and SourceKey identify the folder; the server allocates a CN distinct from that object identity, retains the accepted imported ChangeKey, merges the imported and current PCL lineages, and preserves the winning last-modification time. `Applied` and `Conflict` each allocate that successor CN and write one `mapiOnly` hierarchy delta for MAPI replay and notification; only `Applied` adds its CN to the upload `MetaTagCnsetSeen`. `Conflict` returns `Success` without that upload-state advance so Outlook downloads the resolved version. An exact `Duplicate` returns `IgnoreFailure` without allocating a CN or writing a delta. These `mapiOnly` version events do not enter JMAP `Mailbox/changes`. `PidTagHierRev` and `PidTagLocalCommitTime` use the durable folder LMT, while `PidTagLocalCommitTimeMax` remains the separate maximum change time of a top-level child. The complete tuple transition is implemented for an existing canonical reserved folder; atomic imported-tuple preservation during initial custom-folder creation, rename/move of existing custom folders, imported system-folder alias rename/move, and canonical folder mutations outside this MAPI import path remain interoperability gates. This follows [MS-OXCFXICS] sections 2.2.1.2.3, 2.2.1.2.7, 2.2.1.2.8, 2.2.3.2.4.3.1, 3.1.5.3, 3.1.5.6.2.2, and 3.2.5.9.4.3, and [MS-OXCFOLD] sections 2.2.2.2.1.9, 2.2.2.2.1.13, and 2.2.2.2.1.14. |

## Delete, Move, Tombstone, And Recovery Matrix

| Surface | LPE behavior | Cross-protocol outcome |
| --- | --- | --- |
| `RopDeleteMessages` | Soft delete moves the addressed source mailbox membership to canonical `Trash` when it exists. If the source is `Trash` or any descendant of `Trash`, the operation hard-deletes that addressed membership. `RopExpandRow` (`0x59`) remains a table-category ROP and is never interpreted as delete or hard-delete. | JMAP and IMAP observe the same mailbox membership move or removal through canonical `mailbox_messages`, `mail_change_log`, and `tombstones`. |
| `RopHardDeleteMessages` | Hard delete expunges only the addressed source mailbox membership and writes a canonical `mailbox_message` tombstone with the source IMAP UID. | The message disappears from the addressed IMAP folder and from JMAP when no visible memberships remain; MAPI ICS exports the tombstone as a delete. |
| `RopEmptyFolder` / `RopHardDeleteMessagesAndSubfolders` | Whole-folder purge is implemented only for canonical mailbox folders. `RopEmptyFolder` hard-deletes each visible membership in the target folder only. `RopHardDeleteMessagesAndSubfolders` hard-deletes visible memberships in the target folder and its canonical mailbox descendants. Both paths write the same canonical tombstones/change-log entries as scoped hard delete and create canonical recoverable item rows for retained recovery. System-folder deletion is not implemented because system-folder lifecycle needs explicit canonical semantics. | Deleted Items / Trash purges converge through the canonical delete path. No MAPI-local deleted state, dumpster, or recovery store is created. |
| Whole-folder purge retry semantics | A partial `RopEmptyFolder` / `RopHardDeleteMessagesAndSubfolders` response means successful membership deletes have already committed through canonical state, while failed memberships remain visible. Retrying the same operation is idempotent against canonical visibility: already-expunged memberships are not recreated. Retention and legal hold preserve recoverable item state rather than keeping the source folder membership visible. | Operators can use structured logs and `lpe_mapi_folder_purge_*` metrics to distinguish successful, failed, and partial purge attempts. |
| `RopSynchronizationImportDeletes` | `ImportDeleteFlags` with `HardDelete` (`0x02`) maps to canonical hard delete. The response contains exactly `RopId`, `InputHandleIndex`, and `ReturnValue`; this ROP has no `PartialCompletion` field, as specified by [MS-OXCROPS] section 2.2.13.5.2 and [MS-OXCFXICS] section 2.2.3.2.4.5.2. | MAPI-uploaded deletes do not create protocol-local deleted state; IMAP and JMAP see the resulting canonical move or expunge. |

| Folder move/copy | `RopMoveFolder` and `RopCopyFolder` are bounded to user-created canonical mailbox folders. | Recovery / dumpster | Core `LPE` owns canonical `recoverable_items` state and `/api/mail/recoverable-items` browse, restore, and purge APIs.
